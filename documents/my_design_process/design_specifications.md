# 嵌入式Linux视觉AI采集终端 V4.0 全栈代码编写规范提示词
## 整体定位与适用范围
本规范是**嵌入式Linux视觉AI采集终端 V4.0 架构**唯一官方代码编写标准，适用于所有模块开发、BUG修复、架构迭代工作。所有代码必须严格遵循本规范，禁止私自改动架构规则、编码风格、接口定义。
核心目标：实现模块彻底解耦、代码高可维护性、系统高稳定性、硬件交互安全性，完全对齐Linux内核式设计思想。

---

# 第一部分 全局架构与C语言面向对象设计
## 1.1 核心设计原则
所有模块设计必须严格遵循 **SOLID 原则** 和 **OOP 面向对象思维**：
- **SRP（单一职责原则）**：一个模块 = 一个职责 = 一个变更理由
- **OCP（开闭原则）**：通过函数指针表扩展功能，而非修改现有代码
- **LSP（里氏替换原则）**：所有接口实现必须遵守相同的契约
- **ISP（接口隔离原则）**：拆分臃肿接口，使用者只依赖所需部分
- **DIP（依赖倒置原则）**：高层模块依赖抽象，而非底层模块

## 1.2 四层架构层级铁律（不可越界）
从上至下严格单向依赖，禁止跨层直接调用：
1. **Demo应用编排层**：业务流程组装、用户交互、结果展示
2. **业务服务原子层**：采集/AI检测/推流/LCD显示等自治服务
3. **双总线中间层**：事件总线（控制流）+ 数据总线（数据流）
4. **Main系统框架底层**：系统初始化、信号处理、基础组件、自动初始化

**强制规则**：
- Main层仅负责系统级事件，零业务代码，代码定型后极少修改
- 应用层只通过总线与服务交互，不直接调用服务内部函数
- 服务层只通过总线通信，不感知其他服务存在
- 所有硬件访问必须通过平台抽象层，禁止直接调用HAL函数

## 1.3 C语言OOP实现规范
### 1.3.1 封装模式
- 所有模块对外仅暴露公共接口（头文件），隐藏内部实现
- 头文件使用**不透明指针模式**：`typedef struct Foo Foo_t;`，结构体定义仅在.c文件中
- 禁止从模块外部直接访问结构体成员，必须通过getter/setter或公共API

### 1.3.2 继承与多态
- 派生类结构体**必须将基类作为第一个成员**
- 使用 `container_of` 宏从基类指针恢复派生类指针，禁止强制类型转换
- 多态通过**函数指针表（虚函数表）**实现，所有服务必须继承统一的 `service_base_t` 基类

### 1.3.3 回调模式
- 状态变化必须通过已注册的回调函数通知，禁止使用全局变量或轮询
- 回调必须在头文件中定义类型，并明确标注触发上下文（线程/ISR）
- 回调函数必须简短非阻塞，耗时逻辑必须投递到服务内部队列

## 1.4 自动初始化机制
使用编译器section属性实现编译期自动注册，启动时按优先级顺序执行：
| 优先级 | 宏 | 用途 |
|--------|----|------|
| 1 | `INIT_BOARD_EXPORT(fn)` | 板级硬件初始化 |
| 2 | `INIT_PREV_EXPORT(fn)` | 纯软件预初始化 |
| 3 | `INIT_DEVICE_EXPORT(fn)` | 设备驱动注册 |
| 4 | `INIT_COMPONENT_EXPORT(fn)` | 基础组件初始化 |
| 5 | `INIT_ENV_EXPORT(fn)` | 环境初始化 |
| 6 | `INIT_APP_EXPORT(fn)` | 应用服务初始化 |
| 7 | `INIT_SYSTEM_READY_EXPORT(fn)` | 系统就绪标记 |

**规则**：新增模块只需添加对应宏，无需修改main初始化代码。

## 1.5 驱动架构规范
所有硬件驱动必须实现**非阻塞异步模式**：
- 公共API入队后立即返回
- 驱动内部创建独立工作线程处理硬件操作
- 结果和错误通过回调传递
- 应用层无需了解驱动内部线程实现

## 1.6 模块设计检查清单
创建新模块前必须验证：
1. 能否用一句不含"和"字的话描述模块用途
2. 公共API是否最小化且聚焦
3. 是否依赖抽象而非具体实现
4. 依赖项是否可替换以便测试
5. 模块名称是否清晰传达用途

# 第二部分 Clean Code与华为编码规范
## 2.1 函数设计
- **单一职责**：每个函数只做一件事，禁止"验证并发送"类混合函数
- **抽象层级一致**：同一函数内禁止混合高层编排逻辑和底层位操作
- **参数数量**：最多5个参数，超过则组合为配置结构体
- **命令-查询分离**：函数要么执行操作，要么返回信息，不能同时做
- **无副作用**：函数不能产生隐藏的副作用

## 2.2 命名规范
- 名称长度与作用域成正比：局部变量短，全局变量/公共函数长
- 名称必须揭示意图，禁止无上下文的缩写
- 布尔值使用 `is_`/`has_`/`can_`/`should_` 前缀
- 宏：全大写蛇形 `MAX_RETRY_COUNT`
- 类型：PascalCase + `_t` `UartDriver_t`
- 函数：模块_动作 `UartDriver_SendAsync`
- 局部变量：小写蛇形 `retry_count`
- 文件级静态变量：`s_` 前缀 `s_pending_tx_count`
- 全局变量：`g_` 前缀 `g_system_tick`

## 2.3 防御性编程
- **断言**：用于检查内部契约，捕获程序员错误
- **运行时检查**：用于验证外部输入（用户/网络/传感器数据）
- **const正确性**：所有不被修改的指针参数必须加 `const`
- **static限制**：所有非公共函数和变量必须声明为 `static`

## 2.4 错误处理
- 定义项目级统一错误码枚举，禁止使用裸整数
- 错误必须向上传播，禁止静默吞掉错误
- 错误路径必须清理已分配的所有资源
- 使用goto语句实现统一错误清理

## 2.5 表驱动设计
用表查找替代复杂的if-else链和switch语句：
- 错误码到字符串的映射
- 命令分发（命令ID -> 处理函数指针）
- 状态机转换表
- 配置参数校验范围

## 2.6 DRY原则
- 禁止复制粘贴代码，重复逻辑必须提取为共享函数
- 重复的校验模式、资源初始化/清理序列必须统一封装
- 优先使用 `static inline` 而非宏实现泛型操作

## 2.7 头文件规范
- 每个头文件必须有头文件保护宏
- 头文件必须自包含，可独立编译
- 包含路径必须与项目约定一致
- 包含顺序：对应头文件 → 平台/RTOS头文件 → HAL头文件 → 项目模块头文件 → 标准库

# 第三部分 SOLID原则与GoF设计模式C语言实现
## 3.1 SOLID原则C语言实现
### SRP（单一职责）
每个.c/.h文件对只负责一个功能，例如将UART通信和协议解析拆分为两个独立模块。

### OCP（开闭原则）
通过函数指针表和注册模式实现扩展，新增功能只需添加新条目，无需修改现有代码。

### LSP（里氏替换）
函数指针接口的所有实现必须遵守相同的契约（前置条件、后置条件、错误码）。

### ISP（接口隔离）
将臃肿的接口拆分为聚焦的小接口，使用者只依赖所需部分。

### DIP（依赖倒置）
高层模块依赖抽象函数指针接口，而非底层具体实现，通过依赖注入实现解耦。

## 3.2 常用GoF设计模式
### 状态机模式（State）
使用基于函数指针表的状态机，每个状态对应一个处理函数，状态转换自动执行entry/exit动作。

### 观察者模式（Observer）
实现发布-订阅机制，解耦事件生产者和消费者，支持多订阅者。

### 策略模式（Strategy）
通过函数指针在运行时切换算法，适用于可配置的协议、压缩、加密等场景。

### 工厂模式（Factory）
集中管理对象创建，根据运行时条件返回不同的具体实现。

### 单例模式（Singleton）
使用文件作用域静态实例加访问函数实现，启动时完成初始化，避免多线程延迟初始化问题。

### 适配器模式（Adapter）
将现有接口包装为所需接口，用于集成第三方库或遗留代码。

## 3.3 事件驱动架构
- 使用Active Object模式：每个对象拥有独立的事件队列和线程
- 对象之间通过事件投递或发布-订阅通信
- 支持紧急事件LIFO投递
- 天然线程安全，每个对象的状态机在自己的线程中运行

# 第四部分 硬件交互规则
## 4.1 阻塞分析
在调用任何硬件API之前必须验证其阻塞行为：
- 追踪调用链到寄存器级别
- 检查是否有忙等循环、延时函数、超时参数
- 确认HAL库版本和本地修改情况
- 记录API是阻塞还是非阻塞

## 4.2 事件驱动框架合规性
- 应用层到硬件的整个调用链必须是非阻塞的
- 所有阻塞操作必须在驱动内部工作线程中执行
- 结果通过回调投递事件到应用层
- 驱动内部周期性轮询在自身工作线程中运行

## 4.3 平台相关注意事项
### HAL库
- 检查项目使用的HAL版本和本地修改
- 确认weak回调函数是否被重写
- DMA通道/流配置必须与硬件匹配

### 中断处理
- ISR必须极简，仅释放信号量或设置标志
- 禁止在ISR中分配内存或调用复杂函数
- 使用高优先级工作线程模式处理ISR事件，而非直接ISR回调

### CMSIS-RTOS2
- `osThreadNew()` 必须同时提供 `cb_mem`+`stack_mem` 或两者都不提供
- `osSemaphoreNew()` 可在调度器启动前调用
- `osSemaphoreRelease()` 是ISR安全的

### DMA+环形缓冲区
- 使用DMA循环模式实现非阻塞接收
- 利用UART Idle Line检测实现不定长接收
- D-Cache存在时必须无效化对应缓冲区或使用非Cache区域
- 错误发生时必须abort DMA并重新启动

## 4.4 寄存器级验证
编写硬件代码前必须验证：
- 寄存器地址和位域定义正确
- 使用正确的读-改-写序列
- 所有硬件寄存器指针使用 `volatile` 限定符
- 外设访问宽度正确
- 访问前已启用外设时钟
- 引脚复用功能已正确配置

# 第五部分 内存安全与线程安全
## 5.1 堆内存规则
- 使用项目统一的内存分配API，禁止混用不同分配器
- 每次分配必须检查NULL返回值
- 分配与释放必须配对，错误路径必须清理已分配资源
- 释放后指针必须置为NULL
- 大型局部变量使用堆分配，避免栈溢出

## 5.2 线程安全
### 共享资源保护
| 场景 | 保护机制 |
|------|---------|
| 线程<->线程 | 互斥锁/信号量 |
| ISR<->线程 | 临界区/关中断 |
| 读多写少 | 读写锁 |
| 简单标志位 | 原子操作 |
| 数据交换 | 消息队列（推荐） |

### 消息队列模式
优先使用消息队列实现线程间通信，完全避免共享状态：
- 生产者将消息入队后立即返回
- 消费者独立线程从队列取消息处理
- 天然线程安全，无数据竞争

## 5.3 可重入性
- 可重入函数不使用静态/全局可变变量
- 所有状态通过参数传递
- 调用者提供缓冲区而非使用内部静态缓冲区
- 文档明确标注函数的可重入性

## 5.4 栈溢出防护
- 禁止使用递归和变长数组（VLA）
- 线程中大型局部变量使用堆分配
- 测试时监控栈高水位标记
- 栈大小配置为实际用量加足够余量

## 5.5 缓冲区溢出防护
- 始终将缓冲区大小与指针一起传递
- 复制操作前验证输入长度
- 使用 `memcpy`/`snprintf` 等带大小的函数
- 禁止使用 `strcpy`/`sprintf` 等不安全函数

# 第六部分 项目专属架构规范
## 6.1 双总线通信规范
### 事件总线
- 定位：系统控制中枢，负责命令下发、状态通知
- 线程模型：多线程发布，单线程串行分发
- 回调规则：仅允许引用计数增减、数据入队，禁止任何耗时操作
- 事件分工：Main层发系统级事件，应用层发业务级事件，服务层发状态事件

### 数据总线
- 定位：轻量通知通道，仅传递帧句柄（钥匙）
- 不存储实体数据，不管理帧生命周期
- 推送模式下同步执行所有消费者回调
- 拉模式下仅发送新帧就绪唤醒信号

## 6.2 FrameLink数据链路层规范
- 定位：全局公共帧仓库，唯一存储原始图像数据
- 角色：唯一生产者=采集服务，多消费者=所有视觉服务
- 核心机制：静态内存池 + 原子引用计数 + 帧状态标记
- 帧生命周期：分配→生产→分发→消费→引用归零→回收
- 空闲帧判定：必须同时满足 `ref_count == 0` 且 `bus_deliver_state == 0`

## 6.3 推送模式标准时序
1. 生产者取空闲帧 → `ref_count = 1`
2. 填充数据 → 标记待认领 → 推送数据总线
3. 消费者回调 → `ref_count += 1` → 清除待认领标记 → 入队
4. 所有回调完成 → 生产者释放初始引用 → `ref_count = 消费者数量`
5. 消费者处理完成 → `ref_count -= 1`
6. 引用归零 → 链路层自动回收

## 6.4 拉模式规范
- 数据总线仅发送新帧就绪唤醒信号，不推送帧句柄
- 消费者收到唤醒后主动调用 `frame_link_get_latest_frame()` 拉取最新帧
- 自动舍弃所有积压过期帧，保证实时性
- 底层引用计数、内存池机制完全不变

## 6.5 服务生命周期规范
- 所有服务必须实现统一的7个生命周期接口：init/start/pause/resume/stop/deinit/event_handle
- 服务状态由自身自治管理，外部仅通过事件总线发送控制命令
- 服务内部自建工作线程，隔离所有耗时业务逻辑

# 第七部分 代码风格与格式规范
## 7.1 基本格式
- 行宽：最多80列，超过则在逻辑断点处换行
- 函数长度：最多80行，超过则提取子函数
- 嵌套深度：最多4层，使用提前返回减少嵌套
- 大括号：所有if/for/while/do-while必须使用大括号，即使只有一行
- 固定宽度整数：必须使用 `stdint.h` 的 `uint8_t`/`int32_t` 等类型，禁止使用原生int/short/long

## 7.2 注释规范
- 注释解释"为什么"，而非"是什么"
- 公共函数必须有完整Doxygen格式注释
- 头文件对外接口注释必须零歧义，外部开发者仅通过头文件即可使用
- 删除所有被注释掉的代码，使用版本控制保存历史
- 修改代码时必须同步更新注释

## 7.3 其他规范
- 所有数字字面量（除0、1外）必须定义为宏
- 非void函数的返回值必须检查或显式标注忽略
- 位操作仅用于无符号整数类型
- 禁止使用魔法数字，所有常量定义为宏
