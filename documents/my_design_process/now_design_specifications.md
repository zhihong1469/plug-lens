# 第一阶段专属：嵌入式Linux视觉项目 工程设计规范 + 代码编写全量规范
## 前置核心定位（第一阶段铁律，全篇遵守）
**第一阶段目标**：只做【Linux 6ULL平台功能闭环、BUG修复、多线程稳定性加固、优雅退出全链路打通、业务功能开发】
**第一阶段绝对红线**：
1. 不做跨平台改造、不拆device底层、不做OSAL抽象层
2. 不替换`malloc/free`、不改动态内存逻辑
3. 不删除/重构现有全局上下文`g_app_ctx`等全局结构体
4. 不改动现有多线程锁架构、不打乱已梳理好的锁层级
5. 不重构现有六层分层、不改动已有工程目录骨架
6. 只增量开发、只局部BUG修复、不推倒重来、不大规模重构

# 一、工程目录结构规范（固定不变，禁止私自改动）
## 1. 沿用现有实际目录，禁止新增/删除核心层级
只允许在已有目录内新增`.c/.h`，**不准改动现有目录架构、不准调整目录层级**。
固定目录职责固化：
```
project/
├── Makefile / Makefile.build  # 全局编译规则，禁止随意改顺序、链接依赖
├── common/                    # 通用基础组件：日志/队列/内存池/线程/工具/配置/插件框架
├── bsp/                       # 板级底层，第一阶段仅存放Linux平台相关底层适配，不扩平台
├── plugins/                   # 插件专区：应用插件、硬件device插件、业务服务插件
├── src/                       # 核心架构：入口、总线、状态机、链路、业务服务、device抽象接口
├── documents/                 # 文档、问题记录，代码不往里放
├── output/                    # 编译输出、可执行文件、编译数据库，禁止手写代码放入
└── third_lib/                 # 第三方库，仅存放、不修改源码
```

## 2. 目录文件编写规则
1. 所有模块严格遵循 **inc头文件、src源文件** 分离结构
2. 头文件只放：类型定义、宏、函数声明、外部引用；**禁止放函数实现、全局变量定义**
3. 模块内部私有函数/变量 必须加 `static` 隐藏，不对外暴露
4. 禁止跨目录随意`#include`，只允许依赖上层规范依赖链

---

# 二、Makefile 强制编译&链接规范（一字不准乱改）
## 1. 子目录编译顺序（固定死）
编译顺序**不可调换**：
```
common → plugins → src
```
SUBDIRS 严格固定：`SUBDIRS := common plugins src`

## 2. 静态库链接顺序（强制规则）
GCC 会丢弃未使用库，链接顺序永久固定：
```
-lplug 插件库 → -lcom 公共库 → -pthread 线程库
```
LDFLAGS 严格遵循现有格式，**不准随意增减库、调换顺序**。

## 3. Makefile 开发约束
1. 只允许微调编译选项、宏定义，**不准改动目录编译顺序、链接依赖顺序**
2. 新增模块只需在对应子目录Makefile加入编译文件，不改动顶层架构
3. 保留头文件自动依赖检测机制，不删除该特性

# 三、整体分层架构规范（沿用原有六层，禁止破坏）
## 固定六层分层 & 单向依赖（第一阶段严禁跨层逆向调用）
1. **入口层 App**：main.c、程序生命周期、全局初始化/销毁、信号总控、资源总收口
2. **应用交互层 DemoApp**：主线程select循环、按键命令解析、用户交互
3. **业务服务层 Service**：采集/AI检测业务逻辑、状态机绑定、业务调度
4. **链路管理层 FrameLink**：帧池管理、子线程创建销毁、队列同步、帧流转
5. **硬件抽象层 device**：硬件接口定义、V4L2设备封装、缓冲区管理
6. **基础组件层 Base**：事件总线、数据总线、全局/模块FSM、日志、队列、锁、线程工具

## 分层强制规则
1. 只能**上层调用下层**，禁止下层调用上层、禁止跨层跳跃调用
2. 业务层严禁直接调用Linux原生系统调用，必须走现有封装接口
3. 所有硬件操作、线程创建、总线调用 严格按分层走，不准裸写系统调用

---

# 四、双总线 & 状态机 设计规范（第一阶段严格沿用）
## 1. 双总线强制规则
- **事件总线 EventBus**：只传控制事件、状态变更、指令通知，**不传大数据**
- **数据总线 DataBus**：只承载视频帧大数据、内存池管理、引用计数、自动回收
- 禁止用全局结构体裸传业务数据，所有跨模块数据流转必须走双总线

## 2. 全局+模块双层状态机规范
1. 所有业务模块必须挂载 `ModuleFSM`，受 `GlobalFSM` 统一管控
2. 状态迁移只允许在状态机内部流转，禁止业务代码裸改状态值
3. 硬件启停、资源申请/释放、线程启停 只能在状态机回调中执行
4. 已修复的死锁规避逻辑 **不准改动**：
   - 全局FSM：先拷贝模块列表到临时数组、**锁外执行回调**
   - 模块FSM：**解锁后再调用上层状态回调**

---

# 五、多线程 & 互斥锁 完整强制开发规范（第一阶段核心重点）
## 1. 线程架构固定（不准新增线程层级、不准乱开线程）
- 主线程：main → demo_app_run → select多路复用、命令解析、全局生命周期
- 子线程：仅 FrameLink 采集线程，由FrameLink内部创建/销毁
- 第一阶段**禁止随意新建pthread子线程**，确需新增必须评审锁层级

## 2. 现有锁层级固定（禁止改动现有锁结构、禁止新增反向嵌套）
已固化安全锁层级：
- FrameLink：pool_lock、queue_lock（底层无嵌套）
- CaptureSrv：ctx->lock → 内部ModuleFSM锁
- EventBus：ctx->rwlock 保护订阅列表
- DataBus：ctx->lock / ctx->rwlock / item->ref_lock 三级分层
- ModuleFSM：ctx->lock
- GlobalFSM：ctx->lock
- 通用组件：日志锁、队列锁

## 3. 锁使用铁律（强制遵守）
1. **锁层级单向**：只允许高层锁 → 低层锁，**禁止低层锁反向拿高层锁**，禁止循环嵌套
2. **锁内只做极简操作**
   - ✅ 允许：变量拷贝、状态读取、简单赋值
   - ❌ 禁止：锁内调用其他模块API、锁内耗时操作、sleep、IO阻塞
3. **所有加锁必须成对解锁**，任何分支不能出现漏解锁
4. 多线程共享变量必须加 `volatile` 修饰，防止编译器优化不可见

## 4. 回调函数锁规范
1. EventBus 回调在**读锁上下文**中执行：
   - 允许：简单日志、读取数据总线数据
   - 禁止：在回调里订阅/取消订阅、拿其他高层锁、发布事件嵌套死锁
2. FSM 事件处理回调：逻辑轻量化，不做阻塞、不做耗时业务

## 5. 新增代码锁开发检查清单
新增任何带锁/多线程代码，必须自查3条：
1. 是否形成反向锁嵌套？
2. 是否在锁内调用其他模块API？
3. 是否在回调里持有锁又去申请其他锁？

# 六、优雅退出 & 信号处理 & 资源回收 规范
## 1. 全局退出唯一入口
- 唯一退出标志：`g_app_ctx.app_running`
- 唯一触发函数：`app_trigger_soft_exit()`
- 禁止在业务层私自置位退出标志、私自写管道

## 2. 信号处理规范（禁止修改现有逻辑）
1. 只注册 `SIGINT(Ctrl+C)`、`SIGTERM`
2. 信号处理函数**极简实现**：仅置退出标志、写退出管道
3. 信号函数内**禁止调用**：日志、malloc、pthread、文件操作、非信号安全函数

## 3. 阻塞接口规范
1. 主线程 `select`、子线程 `poll` 必须同时监听：**业务fd + 全局退出管道fd**
2. 所有阻塞IO必须设超时，兜底轮询检查退出标志
3. `EINTR` 信号中断必须规范处理，不能直接continue跳过退出判断

## 4. 固定退出销毁时序（严格按顺序，不准乱序）
```
1. 停止应用交互层 demo_app_deinit()
2. 停止业务服务层 capture_srv/face_detect_srv 销毁
3. 停止帧链路层 frame_link 销毁、等待子线程join回收
4. 销毁全局/模块状态机
5. 销毁事件总线、数据总线
6. 日志、工具组件反向销毁
```

## 5. 资源管理铁律
1. 谁打开谁关闭、谁申请谁释放、谁创建线程谁join等待
2. 所有fd、pthread、互斥锁、内存、总线item 必须 **init/deinit 成对接口**
3. 禁止句柄泄漏、文件fd泄漏、线程僵尸、内存泄漏

# 七、C语言代码编写统一规范（第一阶段全员遵守）
## 1. 编程语言标准
固定使用 **C99 标准**，不使用C11及以上特性，不使用GNU私有扩展语法。

## 2. 命名规范
- 宏定义：全大写+下划线 `MAX_FRAME_BUF_NUM`
- 结构体/枚举/句柄：大驼峰+后缀 `VideoHalHandle_T`
- 全局/模块函数：模块前缀_小驼峰 `frame_link_create`
- 模块内部静态变量/函数：以下划线开头 `_thread_ctx`

## 3. 头文件规范
1. 必须加头文件保护 `#ifndef ... #define ... #endif`
2. 禁止在头文件定义全局变量、禁止函数实现
3. 外部依赖头文件只在`.c`引入，`.h`尽量少嵌套依赖

## 4. 函数编写规范
1. 单一职责，一个函数只做一件事
2. 入参做合法性校验，统一返回错误码
3. 尽量单一出口，便于资源统一回收
4. 函数行数控制，禁止超长函数
5. 对外接口必须写完整注释：功能、入参、返回值、注意事项

## 5. 变量规范
1. 尽量减少全局变量，新增功能优先用模块静态上下文
2. 多线程共享变量加 `volatile`
3. 统一使用标准固定类型 `uint8_t uint16_t int32_t`，不用原生int/short

## 6. 日志规范
1. 统一使用封装`log.h`接口，禁止直接`printf`
2. 分级打印：LOG_E/LOG_W/LOG_I/LOG_D，按场景分级
3. 关键流程、状态迁移、错误返回 必须打日志

---

# 八、新增模块/新增功能 开发流程规范
1. 先在现有目录框架下确定归属层级，不跨目录乱放
2. 先写`.h`接口声明，再写`.c`实现
3. 若涉及多线程/锁：先画锁层级，确认无死锁再编码
4. 若涉及状态机：先定义状态枚举、状态迁移表，再写回调
5. 开发完成必须自测：正常运行、Ctrl+C退出、资源回收、无死锁

---

# 九、BUG修复规范
1. 只**增量局部修复**，不推翻原有架构、不重写整个模块
2. 先分析根因、打印日志、GDB堆栈定位，再改代码
3. 修复后保留原有编码风格、命名、注释规范
4. 修复点做简要备注，记录现象+根因+修复逻辑
