# 三、纯C-OOP 强制编程规则（硬件设备专属）
1. **基类结构体**：仅存放公共属性 + `const` OPS指针（固定首位），无硬件私有参数。
2. **OPS操作表**：必须`const`修饰，存入只读段，函数首参数为基类指针。
3. **子类结构体**：**基类必须作为第一个成员**，存放Linux文件描述符、设备节点、缓冲区等私有参数。
4. **类型转换**：向下转型**强制使用`container_of`宏**，禁止裸指针强转。
5. **对外接口**：统一封装通用接口，内置空指针/OPS校验，禁止外部直接访问`me->ops`。
6. **信息隐藏**：子类实现函数全部`static`修饰，对外隐藏私有细节。

# 四、【新增】代码块边界 铁律规则（核心要求）
## 1. 文件边界规则（绝对禁止混用）
- **.h 头文件**：仅声明**结构体、枚举、宏、对外函数接口**，不实现任何函数逻辑。
- **.c 源文件**：仅实现**函数逻辑、私有变量、OPS表实例**，不暴露私有成员。
- 一个功能模块 = 1个`.h` + 1个`.c`，禁止多模块揉在一个文件。

## 2. 头文件边界规则
- 必须加**头文件保护宏**：`#ifndef _XXX_H_ #define _XXX_H_ #endif`。
- 仅包含本文件依赖的头文件，**禁止冗余包含**。
- 私有结构体/函数**绝不声明在.h中**，仅在.c内定义。

## 3. 函数边界规则
- **单一职责**：一个函数只做一件事，禁止超大函数、多功能糅合。
- **访问权限**：对外函数声明在.h，私有函数加`static`，仅本文件可见。
- **参数规范**：对象操作函数首参数固定为`xxx_base_t *me`，携带上下文。
- **错误处理**：统一返回`int`错误码，`0=成功，负数=失败`。

## 4. 结构体边界规则
- 基类/子类结构体严格分离，**不混用成员**。
- 公共成员放基类，私有成员放子类，禁止跨结构体访问。
- OPS表仅存放函数指针，不存放数据成员。

## 5. 目录边界规则（项目固定）
- 基类/通用接口：`src/device/inc/xxx_base.h` + `src/device/xxx_base.c`
- 子类实现：`plugins/device_plugins/xxx_sub/`
- 业务服务：`src/service/xxx_srv/`
- 数据链路：`src/link/frame_link/`
- 应用层：`src/app/`

## 6. 架构边界规则
- Device层：仅操作硬件/系统调用，**无业务逻辑**。
- Link层：仅管理队列/内存池，**无业务、无硬件**。
- Service层：仅粘合组件，**不操作硬件寄存器**。
- App层：仅流程编排，**不写业务细节**。

# 五、方案三架构 协作强制规则
1. **状态机**：每个服务内置`module_fsm`，仅管理自身状态（IDLE/INIT/RUNNING/ERROR/STOP），无全局状态联动。
2. **服务接口**：所有服务统一暴露4个接口：`xxx_srv_init()`/`start()`/`stop()`/`deinit()`。
3. **总线通信**：
   - 视频帧/大数据 → **数据总线 + frame_link**
   - 事件通知/状态变化 → **事件总线**
4. **服务协作**：无直接函数调用，全靠总线订阅/发布解耦。
5. **流程编排**：固定启停顺序直接写在`app`层，简洁直观、方便调试。
