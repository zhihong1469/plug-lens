# 嵌入式Linux 纯C-OOP visionAI项目 架构约束提示词 V4.0
## 定位说明
适用于本项目所有服务模块、总线层、框架层、Demo应用层代码生成与架构评审；**严格遵循本次敲定的四层架构、生命周期规范、事件总线解耦原则、C-OOP强制规范、模块隔离铁律**，后续所有代码/架构分析/接口设计必须无条件遵守本提示词约束。

---

## 一、整体四层架构严格划分（不可越界、不可跨层直接依赖）
### 层级定义（从上至下）
1. **Demo App 业务编排层**
2. **业务服务原子层**（采集/AI检测/推流等所有业务服务）
3. **双总线中间层**（事件总线 + 数据总线，全局唯一通信中枢）
4. **Main 系统框架底层**（程序入口、系统基建、信号处理、initcall调度、底层资源兜底）

### 层级铁律
1. 下层不感知上层，上层可依赖下层标准接口，**禁止跨层直接函数调用、禁止跨层共享全局变量**；
2. 所有模块间、层级间通信**只允许走事件总线/数据总线**；
3. Main层零业务代码、零服务感知、零业务编排；
4. Demo App层只做业务组装、用户交互、结果订阅，不触碰系统底层基建；
5. 业务服务层只自治自身业务逻辑，不感知Main、不感知Demo App、不依赖外部全局句柄。

---

## 二、Main 系统框架层 职责与约束
### 核心职责
1. 日志初始化、终端模式配置、信号捕获（SIGINT/SIGTERM/SIGABRT/SIGSEGV）；
2. 全局优雅退出管道 `exit_pipe`、系统运行标记 `app_running` 初始化与维护；
3. 全局唯一**事件总线、数据总线**底层初始化；
4. 调用 `do_initcalls()` 执行所有 `MODULE_INIT` 自动注册模块；
5. 主线程仅 `pause()` 休眠等待信号，零CPU占用，不执行业务循环；
6. 程序退出时统一兜底销毁总线、恢复终端、释放底层资源。

### 强制约束
1. `app_context_t` 内部静态私有，**禁止全盘对外暴露**，仅保留极限必要底层接口，业务服务严禁直接获取上下文；
2. `exit_pipe` / `app_running` 为Main层私产，不对外暴露、不允许业务模块访问；
3. Main层**不主动调用任何服务init/start/stop接口**，不感知任何业务服务实例；
4. 系统级启停/暂停/关机，仅通过**广播系统事件**下发，由服务自治响应；
5. 不编写任何业务逻辑、不做服务编排、不硬编码任何业务模块依赖。

---

## 三、双总线中间层 职责与约束
### 核心职责
1. 事件总线：系统控制事件、业务命令事件、模块间通知事件的订阅/发布/分发；
2. 数据总线：视频帧、AI推理结果等大块业务数据的零拷贝分发、订阅、缓冲管理；
3. 提供统一等待fd，支持`select`事件休眠，实现零CPU占用阻塞等待；
4. 自身支持`MODULE_INIT`分级初始化，优先级高于所有业务服务。

### 强制约束
1. 总线句柄**不通过全局变量暴露、不通过`get_context`对外派发**；
2. 总线初始化完成后，自动发布 `SYSTEM_CORE_READY` 事件，将双总线句柄打包为事件载荷**依赖注入**给所有订阅服务；
3. 总线不参与业务逻辑、不维护业务状态，只做纯粹通信中转；
4. 接口固定标准化，所有模块统一调用总线标准API，禁止私有改造。

---

## 四、业务服务原子层 纯C-OOP 强制规范
### 1. 继承结构规范
1. 所有业务服务结构体**首个成员必须是 `service_base_t`**，实现纯C向上转型继承；
2. 服务实例私有化静态定义，对外仅暴露基类指针获取接口，隐藏全部私有成员；
3. 无全局业务变量、无外部硬依赖、不依赖`g_`全局总线句柄、不直接访问app_context。

### 2. 生命周期接口强制规范
严格固定 **7个标准接口不可增删、不可缺省**：
```
init / start / pause / resume / stop / deinit / event_handle
```
1. 所有服务必须完整实现`service_ops_t`函数表，**接口为框架统一管理的标准虚函数表**；
2. 若服务不需要某接口（如pause/resume），必须做空实现、返回0，禁止缺省、禁止修改ops表结构；
3. 基类`service_base_t`统一维护服务状态机：
`SRV_STATE_IDLE / INIT / RUNNING / PAUSE / STOP / ERROR`
4. 服务自身全权维护内部状态流转，仅通过`event_handle`响应总线事件驱动生命周期。

### 3. 服务运行约束
1. 服务内部自建工作线程，业务循环自行维护；
2. 线程休眠统一采用`select`监听总线fd + 退出触发fd，零CPU占用；
3. 总线句柄仅通过订阅`SYSTEM_CORE_READY`事件被动注入，存入自身私有成员；
4. 服务只做单一职责业务逻辑，不编排其他服务、不跨服务直接调用；
5. 只通过事件总线收控制命令、通过数据总线收/发业务帧数据。

### 4. 向上/向下转型使用规范
1. 框架层统一使用**基类指针向上转型**管理所有服务，多态调用生命周期接口；
2. 应用层/框架层**禁止随意向下转型**；仅特殊硬件私有扩展场景可谨慎向下转型，业务常规流程严禁使用。

---

## 五、Demo App 业务编排层 职责与约束
### 核心职责
1. 作为可插拔Demo胶水层，组装多个原子服务（采集+AI检测等）；
2. 独立创建工作线程，不阻塞Main主线程；
3. 订阅数据总线业务结果，做日志打印、用户交互、业务流程控制；
4. 监听退出管道与事件总线fd，实现优雅退出与资源自动清理。

### 强制约束
1. 通过`MODULE_INIT`自动初始化创建工作线程，Main完全无感知；
2. 可获取服务基类实例、调用标准生命周期接口，但**遵循事件驱动优先原则**；
3. 不修改底层框架、不篡改总线逻辑、不占用系统私有资源；
4. 仅处理**用户级业务事件**，系统级关机/启停交由总线事件统一广播；
5. 可随时移除/替换，不影响底层框架与原子服务架构。

---

## 六、事件分工规则（严格划分，不可混淆）
### 1. Main框架层 只负责：系统级硬件/框架事件
`SYS_INIT / SYS_PAUSE / SYS_RESUME / SYS_SHUTDOWN`
- 来源：信号捕获、系统异常、底层框架退出；
- 动作：仅广播事件，不直接控制任何服务。

### 2. Demo App层 只负责：用户级业务事件
用户输入、Demo启停、业务模式切换、结果上报等；
- 可发起业务控制事件，不发起系统底层关机以外的框架级事件。

### 3. 业务服务层 只负责：响应事件、自治处理
- 监听系统事件：自动切换自身生命周期状态、启停暂停；
- 监听业务事件：调整业务参数、切换工作模式；
- 不主动越级发布系统级事件。

---

## 七、Initcall 模块自动注册规范
1. 使用 `MODULE_INIT(fn)` 宏实现段式自动注册，放入`.my_initcall`自定义段；
2. 初始化分优先级：**总线层 > 基础组件 > 业务服务 > Demo App**；
3. Main层通过`do_initcalls()`统一遍历执行，无需逐个包含头文件、无需手动初始化；
4. 新增业务模块只需加`MODULE_INIT`，Main代码零修改，实现可插拔架构。

---

## 八、线程与退出机制规范
1. Main主线程：`pause()` 休眠，仅响应系统信号，零CPU占用；
2. Demo/服务工作线程：`select` 监听退出管道fd + 事件总线fd，阻塞休眠；
3. 优雅退出统一链路：
   信号/用户命令 → 触发软退出 → 广播`SYS_SHUTDOWN`事件 → 所有服务订阅后自治stop/deinit → Demo线程检测退出并清理 → Main最后兜底释放底层资源；
4. 多线程各司其职、无忙轮询、无资源竞争耦合，架构天然线程安全隔离。

---

## 九、编码风格与架构设计红线（禁止违反）
1. 禁止业务服务依赖全局`g_data_bus/g_event_bus`；
2. 禁止业务模块直接包含并依赖Main层底层上下文；
3. 禁止修改`service_ops_t`7接口固定结构、禁止接口选填；
4. 禁止跨层直接函数调用、禁止跨层共享全局变量；
5. 禁止在Main层编写任何业务逻辑、服务编排逻辑；
6. 禁止业务服务向外暴露私有结构体、私有成员；
7. 除特殊硬件私有扩展场景，禁止随意向下转型；
8. 所有资源初始化/销毁成对封装，生命周期由基类接口统一约束。
