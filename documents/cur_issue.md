## 二、 从上到下拆解：六层架构在本项目中的具体含义

我们对着你最开始的架构图，一层一层来看，**结合我们现在的代码**。

### 第6层：App层 (应用业务层)
*   **设计目标**：**只做业务组合，不碰底层代码**。就像一个产品经理，只负责说“我要一个采集+显示的功能”，不负责写摄像头驱动。
*   **理想中的工作流**：
    1.  用户按 `s` 键；
    2.  App层往 **Event Bus** 发一个 `EVENT_TYPE_CAPTURE_START` 事件；
    3.  然后就不管了，坐在那里等 **Event Bus** 通知它“帧准备好了”。
*   **当前代码现状** (`plugins/app_plugins/src/demo_app.c` + `src/app/main.c`)：
    *   我们的 `demo_app` 确实在处理用户按键（`s`/`t`/`q`）。
    *   但是，它**没有发事件**，而是直接通过 `extern` 函数调用了 `main.c` 里的 `demo_app_trigger_capture_start()`。
    *   这违反了「App层只能通过总线交互」的铁律。

### 第5层：FSM层 (状态机层)
*   **设计目标**：**系统的大管家**。负责管控整个系统的生命周期（初始化、就绪、运行、降级、关机）。
*   **核心价值**：防止非法操作。比如，你不能在“未初始化”的时候要求“开始采集”，也不能在“运行中”直接“断电”。
*   **当前代码现状**：
    *   我们在 `main.c` 里初始化了 `g_global_fsm`。
    *   但是，**我们完全没有用它**。我们没有注册任何子模块，也没有往它投任何事件。
    *   它就像一个被放在一边的红绿灯，我们直接在路口指挥交通。

### 第4层：Service层 (原子服务层)
*   **设计目标**：提供**独立的原子能力**。比如“采集”、“显示”、“AI推理”都是独立的服务。
*   **核心规则**：服务之间**绝对不能直接调用**。比如“AI服务”想拿帧，不能直接喊“采集服务，把帧给我”，必须通过 **Data Bus** 去拿。
*   **当前代码现状** (`src/service/capture_srv/`)：
    *   ✅ 这一层我们实现得不错！`capture_srv.c` 封装得很好，只提供 `start/stop/get_frame/put_frame`。
    *   ❌ 但是，我们在 `main.c` 里直接 `#include "capture_srv.h"` 并调用它了。这在架构上是“越级”了（App层应该通过FSM/总线，而不是直接调Service）。

### 第3层：Link层 (数据链路层)
*   **设计目标**：**管数据流转，不管硬件**。它负责从HAL拿数据，放到自己的帧池里，维护好队列，防止丢帧。
*   **核心规则**：绝对不修改帧数据，也不做AI推理。
*   **当前代码现状** (`src/link/frame_link/`)：
    *   ✅ 这一层是目前完成度最高的！帧池、队列、采集线程、流控逻辑都在正常工作（从日志里 `index` 循环可以看出来）。

### 第2层：HAL层 (硬件抽象层)
*   **设计目标**：**屏蔽硬件差异**。比如将来你换了一个MIPI摄像头，只需要换一个HAL插件，Link层及以上的代码一行都不用改。
*   **核心规则**：纯函数操作，无状态，不创建线程。
*   **当前代码现状** (`plugins/hal_plugins/video_usb/`)：
    *   ✅ 完美实现。V4L2的所有操作都封装在这里了。

### 第1层：硬件层
*   就是你的USB摄像头 `/dev/video0`。

## 三、 核心难点解析：为什么要有“双总线”？

这是新手最困惑的地方：“我直接函数调用就能搞定的事，为什么要搞个总线？”

我们结合这个项目来理解 **Event Bus (事件总线)** 和 **Data Bus (数据总线)** 的分工。

### 1. Data Bus (数据总线)：运货的高速公路
*   **作用**：传输**大数据**（视频帧、AI结果）。
*   **设计亮点**：**零拷贝**。
*   **在我们项目中的理想流程**：
    1.  **Link层** 从摄像头拿到数据；
    2.  Link层把数据指针 **发布 (Publish)** 到 Data Bus；
    3.  **App层** (或者AI服务层) 从 Data Bus **获取 (Acquire)** 这个指针（引用计数+1）；
    4.  用完后 **释放 (Release)**（引用计数-1）；
    5.  如果没人用了，自动放回帧池。
*   **当前现状**：我们在 `main.c` 里直接 `capture_srv_get_frame`，这相当于“抄近道”，没有经过这个“物流中心”。

### 2. Event Bus (事件总线)：传令兵
*   **作用**：传输**小数据**（指令、状态、报错）。
*   **设计亮点**：**解耦**。
*   **在我们项目中的理想流程**（对比当前流程）：

| 动作 | 当前“硬编码”流程 (耦合) | 理想“总线”流程 (解耦) |
| :--- | :--- | :--- |
| **按 `s` 键** | `demo_app` 直接调用 `capture_srv_start()` | `demo_app` 发布 `EVENT_TYPE_CAPTURE_START` 事件<br>-> **Event Bus** 广播<br>-> `capture_srv` 收到事件，自己启动 |
| **帧就绪** | `main.c` 轮询 `capture_srv_get_frame` | `capture_srv` 发布 `EVENT_TYPE_CAPTURE_FRAME_READY`<br>-> **Event Bus** 广播<br>-> `demo_app` 收到通知<br>-> `demo_app` 去 **Data Bus** 取帧 |

**解耦的好处**：
如果将来我想加一个“AI推理服务”，我不需要改 `demo_app` 的代码，也不需要改 `capture_srv` 的代码。我只需要让“AI服务”也去订阅 Event Bus 的事件、去 Data Bus 取数据就行了。

## 四、 分析当前的遗留问题与隐患

我们来对应日志看问题：

### 问题1：按 `q` 键和 `Ctrl+C` 无法完全退出 (最紧急的表面问题)
**日志现象**：
```text
q
[I] ... Demo App: User requested quit
^C
[I] ... Demo App: Received signal 2, stopping...
^C
[I] ... Demo App: Received signal 2, stopping...
(程序卡住不动)
```
**原因分析**：
我们有两个 `g_running` 变量：
1.  一个在 `main.c` 里 (`static volatile bool g_running`)；
2.  一个在 `demo_app.c` 里 (`static volatile bool g_running`)；
当你按 `q` 时，`demo_app.c` 里的那个变成 `false` 了，但 `main.c` 主循环里检查的是它自己的那个。虽然代码里写了 `extern volatile bool g_running`，但由于变量作用域和 `volatile` 的使用在多文件中很容易出问题，导致主循环没退出来。

### 问题2：架构分层被破坏 (长期隐患)
*   **当前问题**：`main.c` (App层) 直接 `#include "capture_srv.h"` (Service层)。
*   **隐患**：如果我们继续这样写，过不了多久，代码就会变成“一锅粥”。所有层都互相调用，最后改一个地方，全系统都要跟着改。

### 问题3：采集服务停止逻辑可能有隐患
**日志现象**：
你按了 `t` 之后，日志还在打 `Got frame!`。
```text
t[I] ... Main: Got frame! index=2...
[I] ... Main: Got frame! index=3...
```
**原因分析**：
这不一定是Bug（因为我们是在 `main.c` 里轮询的，可能按 `t` 那一刻队列里还有几帧），但这反映了一个问题：我们没有一个**统一的状态视图**。如果有FSM层，我们可以很清楚地知道“现在系统到底是在 RUNNING 还是 STOPPED”。
