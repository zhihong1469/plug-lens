# 一、旧架构：六层双总线+分层状态机（详细拆解）
## 1. 逻辑架构：严格6层（层层独立，重型设计）
这是你最初设计的**工业车规级架构**，每一层都有明确的逻辑边界，追求极致隔离与扩展：

| 逻辑层 | 核心职责 | 对应你原目录 | 为什么是“重型” |
|--------|----------|--------------|----------------|
| 1. 业务应用层 | 主流程、业务编排 | `src/app`、`plugins/app_plugins` | 只做业务，不碰底层 |
| 2. 服务层 | 业务服务抽象 | `src/service`、`plugins/service_plugins` | 封装业务逻辑，提供统一接口 |
| 3. 链路层 | 数据流转控制 | `src/link` | 负责模块间数据传递，复杂路由 |
| 4. 双总线层 | 数据/事件统一通信 | `src/bus`（data_bus+event_bus） | 工业级通信机制，冗余设计 |
| 5. HAL层 | 硬件抽象隔离 | `src/hal`、`plugins/hal_plugins` | 隔离USB/MIPI/网络摄像头差异 |
| 6. 分层状态机层 | 全局+模块状态管理 | `src/fsm`（global_fsm+module_fsm） | 车规级状态控制，异常处理复杂 |

## 2. 物理架构：严格6层（一一对应，无内嵌）
你的旧架构**逻辑层=物理层**，每一层都有独立的文件夹和代码，完全物理隔离：
- 每个逻辑层都有自己的`inc`和`src`目录
- 层间通信必须通过严格的接口，不能跨层调用
- 调试时需要逐层排查，定位问题慢，周期长
- 你之前提到的“调试成本极大，项目周期极长”，根源就在这里

# 二、新架构：5层逻辑架构（详细拆解）
## 1. 逻辑架构：5层（收敛冗余，聚焦核心）
新架构从**第一性原理**出发，合并冗余层，保留项目必须项：

| 逻辑层 | 核心职责 | 对应你新目录 | 为什么“合适” |
|--------|----------|--------------|--------------|
| 1. 业务应用层 | 主流程、业务编排 | `src/app`、`plugins/app_plugins` | 不变，只做业务 |
| 2. 组件服务抽象层 | 合并原服务+链路+总线+HAL+FSM | `src/service`、`src/link`、`src/device`（原hal）、`src/bus`、`src/fsm` | **C-OOP核心层**，父类封装公共逻辑，子类实现硬件差异 |
| 3. Linux系统适配层 | 封装V4L2/mmap/ioctl等系统调用 | 内嵌在`src/device`内部 | 薄胶水层，不独立文件夹，减少冗余 |
| 4. 通用工具层 | 日志/队列/池/线程/工具 | `common/` | 完全不变，复用基础组件 |
| 5. Linux内核层 | 驱动/设备节点，提供系统接口 | 无独立文件夹（直接调用） | 不编码，只调用内核接口 |

## 2. 物理架构：4层（内核层不编码，系统适配层内嵌）
这是你最关心的点——**逻辑5层≠物理5层**，物理上你只需要维护4层：
1. **业务层**：`src/app` + `plugins/app_plugins`（物理1）
2. **组件层**：合并原5层逻辑（服务+链路+总线+HAL+FSM）→ 物理2
3. **通用工具层**：`common/`（物理3）
4. **系统适配层**：内嵌在组件层的`src/device`中，不独立（物理3的一部分）
5. **内核层**：不写代码，只调用（无物理层）

# 三、核心差异：为什么新架构更合适？
## 1. 逻辑层变化：从“6层独立”到“5层收敛”
- 旧：6层逻辑，每层都有独立职责，层层隔离，扩展性强但冗余
- 新：**合并5层逻辑到“组件服务抽象层”**，只保留核心业务+组件+工具+系统适配+内核
- 核心：用**C-OOP父类+子类**替代原来的HAL层，实现硬件隔离，同时消除冗余

## 2. 物理层变化：从“6层物理”到“4层物理”
- 旧：每一层都有独立文件夹，物理上完全分离，调试路径长
- 新：**只保留4层物理目录**，系统适配层内嵌在组件层，内核层不编码
- 核心：物理目录不变（只改hal→device），逻辑上合并，调试路径短，效率高

## 3. 最关键的取舍：从“极致扩展”到“最合适”
| 取舍维度 | 旧架构（六层双总线） | 新架构（5层逻辑） |
|----------|----------------------|-------------------|
| 扩展性 | 极致，支持未来所有可能需求 | 足够，满足当前项目+可扩展新设备 |
| 调试成本 | 极高，需要逐层排查 | 低，组件层统一调试，路径短 |
| 开发周期 | 极长，每层都要写大量代码 | 短，合并冗余层，复用现有代码 |
| 维护难度 | 大，层间接口多，文档复杂 | 小，逻辑收敛，接口清晰 |

# 最后给你彻底消除迷茫的总结
1. **旧架构：6层逻辑 = 6层物理**，每一层都有独立文件夹，重型设计，适合车规级大规模项目
2. **新架构：5层逻辑 = 4层物理**，内核层不编码，系统适配层内嵌在组件层，轻量化设计，适合你的视觉采集终端项目
3. **物理目录改动极小**：只需要把`hal`→`device`，其他文件夹完全不变
4. **核心思想**：架构不是越多层越好，而是**逻辑清晰、物理精简、适配项目规模**的才是最好的
