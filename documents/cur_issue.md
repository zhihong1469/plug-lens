# 完整项目上下文&接续开发提示词（可直接复制给任意AI助手无缝接续开发）
## 一、项目基础信息
项目名称：**嵌入式Linux IMX6ULL 视觉AI采集终端**
编译运行环境：正点原子100ask IMX6ULL 开发板、Linux 嵌入式环境、GDBServer 远程调试、V4L2 USB摄像头
工程设计理念：
1. **核心框架固化**：`src/` 底层核心架构、总线、状态机、链路层尽量永久不动；
2. **业务插件化**：所有业务功能、应用层逻辑全部下沉至 `plugins/` 目录，后续新增设备/AI业务仅修改插件；
3. **分层解耦+单向数据流**：双总线单向上行、状态机单向下行管控硬件，层级严格隔离；
4. **强健壮性要求**：多线程安全、锁严格管控、信号安全处理、全链路优雅退出、无全局混乱变量。

## 三、核心系统分层&固定架构规则
### 1. 核心分层（自上而下）
应用交互层 → 业务服务层 → 全局FSM层 → 模块FSM层 → 链路层FrameLink → HAL硬件抽象层 → 物理硬件

### 2. 两大核心核心组件
1. **双总线架构**
- EventBus：系统事件、状态变更、控制事件上行通知；
- DataBus：视频帧大数据缓冲、异步流转；
- 数据规则：**仅允许 服务层→应用层 单向上行**，禁止反向调用耦合。

2. **双层FSM状态机架构**
- Global FSM：全局统一状态管控、模块注册、全局状态决策；
- Module FSM：单个服务/硬件独立状态机；
- 控制规则：**全局FSM单向向下下发控制事件**，严格管控硬件启停、异常降级。

### 3. 多线程模型
- 主线程：select 多路监听（键盘+EventBus+退出管道）、人机交互、业务调度；
- 采集子线程：FrameLink 循环拉取摄像头帧数据、V4L2 poll 阻塞等待；
- 所有线程必须支持**快速唤醒、信号感知、优雅回收**。

## 四、强制编码&锁管理规范（必须严格遵守）
1. **锁设计铁律**
- 禁止**自死锁（同一线程重复拿同一互斥锁）**；
- 禁止**锁嵌套**，若必须嵌套需固定全局锁顺序；
- 锁持有时间最小化：锁内仅做数据拷贝/变量读取，**禁止在锁内调用状态机事件、回调、业务函数**；
- 解决方案：锁内拷贝句柄/数据，锁外执行业务逻辑（已在global_fsm完成修复）。

2. **内存&变量规范**
- 彻底消灭零散全局变量；
- 所有全局资源（总线、状态机、服务、文件描述符）统一收敛至 `common/configs/main.h` 内 `app_context_t` 统一管理；
- 局部业务私有变量使用静态私有上下文，杜绝全局污染。

3. **信号处理强制规范**
- 信号处理函数必须**异步信号安全**；
- 禁止在信号函数内调用日志、IO、锁、内存申请、终端操作；
- 仅允许：写入退出管道、原子变量赋值；
- 采用 `sigaction` 替代老旧 `signal` 函数，兼容多系统。

4. **阻塞IO规范**
- 所有 poll/select 必须添加合理超时；
- 禁止硬件层无限阻塞等待；
- 后续所有阻塞IO必须**同时监听全局退出管道fd**，实现毫秒级唤醒。

## 六、当前残留问题 & 待完善清单
1. **运行中 Ctrl+C 仍存在卡死问题**
根因：摄像头采集子线程 `video_usb.c` 内 poll 仅监听硬件fd，未监听退出管道；按下Ctrl+C后主线程收到管道退出信号，但子线程卡在硬件poll等待，无法及时退出，导致join回收卡死。

2. 退出能力不完整
- 仅主线程实现了Pipe退出监听；
- FrameLink采集线程、HAL层阻塞IO未接入全局退出管道；
- 线程回收逻辑缺少超时兜底、强制回收机制。

3. 资源释放细节待补全
- 硬件流停止、V4L2 缓冲区回收在异常退出场景缺少强触发；
- 锁资源、线程句柄异常场景兜底释放逻辑不足。

## 七、未来核心开发目标：全链路标准化优雅退出
### 最终目标
实现**任意场景零卡死退出**：正常按键退出、Ctrl+C强制退出、kill进程退出、异常崩溃兜底，全部满足：
1. 毫秒级唤醒所有阻塞线程（采集线程、IO阻塞线程）；
2. 按层级有序退出：应用层→服务层→链路层→硬件层；
3. 锁统一解锁、线程正常join、硬件关闭、缓冲区释放、文件描述符关闭；
4. 终端自动恢复、无僵尸进程、无资源泄漏、无残留死锁；
5. 完全基于**全局退出Pipe管道**事件驱动，抛弃轮询flag，架构优雅可扩展。

### 需落地的具体开发任务
1. **FrameLink 层改造**
- frame_link 上下文增加退出管道读端fd；
- 采集线程循环内监听管道事件，收到退出信号立即终止循环。

2. **HAL层 video_usb.c 深度改造**
- 改造 `_video_usb_dqbuf` 的 poll 逻辑，由单fd监听改为**双fd监听**：摄像头fd + 全局退出管道读端；
- 检测到管道可读后，立即返回退出错误，中断硬件阻塞等待。

3. **完善线程安全回收机制**
- 所有子线程统一增加退出管道监听；
- 线程join增加超时控制，超时后启用pthread_cancel兜底；
- 服务停止接口增加强断流逻辑，主动唤醒硬件阻塞IO。

4. **统一异常退出流程封装**
- 完善全局上下文资源销毁顺序；
- 新增异常崩溃兜底钩子，保证基础硬件与终端恢复。

4. 线程卡死优先排查：HAL阻塞IO、无超时poll、未监听退出事件。
