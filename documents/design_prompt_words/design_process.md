# 嵌入式Linux视觉采集终端 架构迭代心路历程笔记
## 前言
记录项目从**V1.0原生朴素架构 → V2.0 六层双总线+分层状态机重型架构 → V3.0 轻量化5层逻辑架构** 完整迭代心路，复盘每一代架构设计初衷、隐性分层、存在缺陷、迭代动机，正视早期无主动逻辑分层思想的设计短板。

---

## 一、架构 V1.0：原生朴素物理架构（初始起手第一版）
### 1. 当时设计思路（原始本心）
最初期无**主动逻辑分层设计思想**，仅从物理文件、功能模块直观拆分目录与代码：
1. 底层单独做 **HAL硬件抽象层**，核心目的只为隔离不同摄像头硬件差异（USB摄像头、MIPI摄像头、网络摄像头）；
2. 独立拆分 **通用工具公共层**，存放日志、队列、工具函数等全项目复用基础组件；
3. 上层直做 **业务逻辑层**，一个业务对应一个独立模块，模块内部自行实现多线程、数据链路流转；
4. 设计初衷：后续最上层应用层，直接调用各业务模块对外暴露的极简接口即可，实现业务解耦。

### 2. 暗含隐性逻辑分层（当时未主动设计，天然隐含）
> 无主动架构设计意识，但物理目录天然形成了简陋逻辑分层

1. **业务应用层**：最上层入口，调用各业务模块接口
2. **业务模块逻辑层**：各独立业务模块（采集、后续预留AI等），内部糅合多线程、数据链路、业务逻辑
3. **硬件抽象HAL层**：屏蔽摄像头硬件类型差异
4. **通用工具公共层**：基础工具、容器、日志等无业务依赖组件

### 3. V1.0 核心致命问题复盘
1. **无主动逻辑分层思想**，误以为「一个文件/一个文件夹对应一个层级」，仅靠物理目录拆分，架构无规范约束；
2. **业务模块内部层次严重混乱**：服务逻辑、线程管理、数据链路、业务处理全部糅合在同一模块内，职责边界完全模糊；
3. **多线程架构无统一规范**：各模块自建线程、自建队列、裸共享变量通信，极易引发死锁、数据竞争、资源泄漏；
4. **扩展性极差**：新增模块成本极高，例如新增AI人脸检测模块、LCD显示模块时，无法复用现有架构规范，代码容易散乱堆砌；
5. **无统一通信机制**：模块间直接硬依赖、函数互调，后期维护、迭代、模块替换难度极大。

### 4. 迭代动机
深感V1.0分层不干净、职责混杂、多线程风险高、新增模块极易架构腐化，必须升级为规范化重型架构，约束代码写法与分层规则。

## 二、架构 V2.0：六层双总线 + 分层状态机 重型架构（第二版）
### 1. 设计初衷
为解决V1.0无规范、层次混乱、线程杂乱、扩展性差的问题，直接采用**工业车规级重型架构**欲一步到位，强行做标准化分层、统一模块通信、全生命周期管控：
1. 严格定义六层独立逻辑分层，做到职责单一、单向依赖；
2. 引入**事件总线+数据总线**双总线机制，实现模块间解耦、零拷贝数据流转、异步事件通知；
3. 引入**模块状态机+全局状态机**分层状态机，统一所有模块生命周期、状态迁移、异常处理；
4. 规范插件化思想，拆分核心骨架与业务实现，预留大量扩展能力，为未来多外设、多业务、多平台移植做铺垫。

### 2. V2.0 逻辑分层 & 对应物理分层
#### 逻辑六层（严格独立）
1. 应用业务层
2. 全局/子状态机层
3. 业务服务层
4. 数据链路层
5. 硬件抽象HAL层
6. 通用基础组件层

#### 物理分层
逻辑六层 **完全一对一映射物理文件夹**，每一层独立目录、独立编译、严格隔离，形成「逻辑层=物理层」的重型架构形态。

### 3. V2.0 核心问题复盘
1. **过度设计、杀鸡用牛刀**：车规级重型架构适配大型多设备、多业务、复杂工控场景，本项目仅为单视觉采集+AI推理线性业务，大量预留功能、复杂机制实际用不到；
2. **架构疑似冗余**：分层过细、层级职责部分细节冗余嵌套；
3. **开发&调试成本极高**：层级过多、总线+状态机嵌套复杂，问题定位链路长，项目迭代周期被大幅拉长；
4. **形式主义分层**：为了凑层数、凑架构规范而拆分目录，部分层级无实际业务价值，徒增代码量与维护成本。

### 4. 迭代动机
意识到**没有最好的架构，只有最合适的架构**，放弃盲目追求极致扩展的重型设计，从第一性原理出发，做兼并精简、去冗余、适配项目实际体量的重构。

## 三、架构 V3.0：轻量化5层逻辑架构（最终改革版）
### 1. 设计初衷
保留V1.0、V2.0中的优秀设计思想（硬件隔离、插件化、通用组件、总线解耦），剔除所有冗余分层、无效预留、过度设计；
遵循**逻辑分层规范约束、物理目录精简复用**原则，不强行让逻辑层对应物理文件夹，合并冗余层级、收敛架构复杂度。

### 2. V3.0 逻辑分层 & 对应物理分层
#### 逻辑五层（标准逻辑架构）
1. **业务应用层**
2. **组件服务抽象层**（合并原服务/链路/总线/FSM/硬件抽象能力，C-OOP唯一落地层）
3. **Linux系统能力适配层**（薄封装内核接口，内嵌不独立建目录）
4. **通用工具公共层**
5. **Linux内核原生层**（仅调用，不编写业务代码、无物理目录）

#### 物理四层（实际维护的代码层级）
1. 业务物理层：`src/app` + `plugins/app_plugin`
2. 组件核心物理层：合并原service/link/bus/fsm/device（原hal）+ 对应plugins插件目录
3. 通用工具层：`common` 目录完全复用
4. 系统适配层：内嵌在组件层内部，不单独拆分物理目录

> 内核层为系统原生提供，不参与物理代码目录管理。

### 3. V3.0 架构核心取舍亮点
1. 正视早期无逻辑分层思想的短板，确立**先逻辑规则、后物理目录**的设计原则；
2. 取消冗余独立HAL层，更名`hal`为`device`，用C语言OOP父子类天然实现硬件设备差异隔离，摒弃裸机式重型硬件抽象；
3. 精简分层，把原V2.0多层冗余逻辑收敛到**组件服务抽象层**，物理目录不做大拆大改；
4. 保留双总线、状态机、插件化、多线程安全等核心刚需能力，仅做轻量化裁剪，不彻底推翻重来；
5. 架构适配项目体量，兼顾可维护性、可扩展性、开发调试效率，拒绝超前冗余堆砌。

## 四、三代架构 逻辑/物理分层 总对比汇总
| 架构版本 | 逻辑分层数 | 物理分层特征 | 核心设计特征 | 核心痛点 |
|----------|------------|--------------|--------------|----------|
| V1.0 初始朴素架构 | 4层（隐性无主动设计） | 物理简单拆分，无规范约束 | 按功能直观分目录，模块内糅合线程/链路/业务 | 无逻辑分层思想、层次混乱、多线程易死锁、扩展性差 |
| V2.0 六层双总线架构 | 6层（主动规范设计） | 逻辑层与物理层一一对应 | 重型车规级架构、全分层、双总线、分层状态机 | 过度设计、冗余套娃、调试成本高、杀鸡用牛刀 |
| V3.0 轻量化重构架构 | 5层（主动收敛设计） | 逻辑5层、物理仅4层，内核层无物理目录 | 逻辑定规则、物理复用旧目录，合并冗余层级、OOP设备抽象 | 无明显短板，适配本项目体量，平衡扩展与效率 |

## 五、架构迭代总结感悟
1. 架构初期**没有主动逻辑分层思想**，仅靠物理文件直观拆分，必然导致后期代码混乱、边界模糊；
2. 盲目套用工业级重型六层架构，忽视项目实际体量，只会造成过度设计、开发负重；
3. 优秀架构不是层数越多越好，而是**逻辑分层定规范、物理目录顺习惯**，按需分层、去冗余、留刚需；
4. Linux应用层架构核心原则：复用内核硬件抽象，不重复造轮子，用C-OOP做业务设备抽象，而非裸机式厚重HAL分层。
