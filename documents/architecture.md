## 二、核心名词概念统一（消除歧义，全局通用）
> 以下定义为项目强制标准，所有沟通、编码、文档统一使用。
1. **OSI数据链路层**：Linux内核驱动范畴，对应项目 `drv/` 目录，负责USB/以太网等硬件帧协议，**与应用层架构完全隔离**。
2. **data_bus 数据总线**：应用层内部**原始数据流转发管道**，仅做帧句柄路由、分发，不修改数据、不管理内存。
3. **frame_link 应用数据链路**：应用层自定义链路组件，当前主力负责**视频帧生命周期管理**（内存池、引用计数、帧回收），**预留自定义应用协议封包/解包能力**；**不承载图像格式转换、编解码逻辑**。
4. **event_bus 事件总线**：应用层控制流通道，传输指令、状态、异常、启停等控制类消息。
5. **initcall 自动初始化**：结合自定义链接段实现编译期函数注册，按优先级自动执行初始化，仿照Linux内核机制。
6. **抽象基类**：`src/base/` 下通用基类（摄像头/AI模型/LED），高层模块仅依赖抽象，不依赖具体硬件实现。
7. **插件体系**：分为三类插件，依托自动初始化机制加载，是业务与硬件适配的主要载体。
8. **fsm 状态机**：全局/模块状态机组件，归属中间枢纽层，当前代码预留、暂未启用。

## 三、逻辑分层架构（设计视角 · 核心标准）
> 划分依据：**功能职责、运行时依赖、数据流/控制流**；仅包含运行时业务代码，不含文档、编译产物；**自上而下单向依赖，禁止跨层直连**。
共分为**四层逻辑架构**，为项目架构设计、接口评审、模块划分的核心依据。

### 层级总览（从上至下）
`应用编排层` → `业务服务原子层` → `中间枢纽层` → `系统基建层`

### 3.1 第一层：应用编排层（最顶层）
#### 核心职责
1. 全系统业务流程组装、全局启停调度、资源收尾；
2. 预留人机交互、UI界面、用户指令解析扩展能力；
3. 本项目现状：无人机交互，仅实现开机自启、异常处理、安全退出，代码极简。

#### 包含模块
`plugins/app_plugins` 下所有应用代码（`app.c`、`demo_app.c`）

#### 依赖规则
- 仅依赖：中间枢纽层、业务服务层对外抽象接口；
- 禁止：直接操作硬件、调用底层基建私有接口、跨层直连服务内部函数。

#### 变更频率
高（业务流程、交互需求迭代时修改）

### 3.2 第二层：业务服务原子层（核心业务载体）
#### 核心职责
1. 按**单一职责**拆分自治服务，每个服务独立线程运行；
2. 服务间零直接调用，所有交互统一通过中间枢纽层完成；
3. 承载采集、AI推理、流媒体推流、存储、硬件指示灯等核心业务。

#### 包含模块
1. 业务服务插件：`plugins/service_plugins`（采集服务、人脸检测服务、RTSP推流服务）
2. 基础硬件能力插件：`plugins/base_plugins`（USB相机、MNN AI模型、SD存储、LED控制器、图像辅助组件）

#### 依赖规则
- 仅依赖：中间枢纽层、系统基建层、抽象基类；
- 所有硬件操作通过**基类+插件**间接实现，禁止直写硬件逻辑。

#### 变更频率
中（功能迭代、算法优化、业务逻辑调整时修改）

### 3.3 第三层：中间枢纽层（解耦核心，架构枢纽）
#### 核心职责
全局控制流转发、数据流路由、帧资源管理、状态管控，是全模块通信的唯一中转层。

#### 包含四大子模块 & 细分职责
1. **event_bus 事件总线**
   - 传输内容：控制命令、服务状态、异常告警、启停指令（小数据控制流）；
   - 模型：多生产者、单线程串行分发，回调轻量化非阻塞。
2. **data_bus 数据总线**
   - 传输内容：摄像头原始YUYV帧等裸数据流；
   - 规则：**纯转发、纯路由**，不修改数据、不分配/释放内存、不做格式转换。
3. **frame_link 应用数据链路**
   - 核心能力1（已落地）：全局视频帧生命周期管理（静态内存池、原子引用计数、超时自动回收）；
   - 核心能力2（预留扩展）：自定义应用协议封包、解包、帧校验；
   - 红线：**严禁承载色彩空间转换、JPEG/H264编解码等计算逻辑**。
4. **fsm 状态机组件**
   - 包含：全局状态机、模块独立状态机；
   - 现状：代码预留，当前项目暂未启用，后续复杂状态流转场景使用。

#### 依赖规则
- 仅依赖：系统基建层；
- 是上层所有模块通信的唯一入口，禁止上层绕过本层直接通信。

#### 变更频率
极低（架构定型后，仅BUG修复、协议扩展时修改）

### 3.4 第四层：系统基建层（最底层 · 项目基座）
#### 核心职责
提供全项目通用基础能力、抽象接口、编译期初始化机制，**无任何业务逻辑**，是所有上层模块的依赖源头。

#### 包含模块
1. `common/` 全局公共组件：日志、内存适配、内存池、线程封装、线程安全队列、守护进程、系统时间、SD挂载、网络检测、工具函数、图像基础处理；
2. `src/base/` 抽象基类：摄像头基类、AI模型基类、LED硬件基类（依赖倒置核心）；
3. `src/app/` 系统入口、信号捕获、全局安全退出逻辑；
4. `initcall` 自动初始化组件 + `custom_sections.lds` 自定义链接脚本。

#### 依赖规则
- 本层**不依赖任何上层模块**；
- 全项目所有模块必须依赖本层基础接口。

#### 变更频率
极低（基建定型后，几乎不修改）

## 四、物理分层架构（工程视角 · 编译&目录标准）
> 划分依据：**磁盘目录、编译单元、编译顺序、编译方式、版本管理**；用于Makefile编写、文件维护、任务分工、编译调试。
整体分为**五大代码物理层 + 附属资源目录**，编译顺序：**从底层到顶层依次编译**；`drv/` 内核驱动为完全独立编译单元。

### 4.1 物理层1：全局基建目录（最先编译）
- 对应逻辑层：系统基建层
- 包含目录：`common/`、`src/`、顶层`Makefile`、`Makefile.build`、`custom_sections.lds`
- 特性：全项目基础依赖，架构定型后极少改动。

### 4.2 物理层2：基础硬件&能力插件（第二层编译）
- 对应逻辑层：业务服务原子层（硬件适配子类）
- 包含目录：`plugins/base_plugins`
- 特性：继承底层抽象基类，实现具体硬件/算法能力。

### 4.3 物理层3：核心业务服务插件（第三层编译）
- 对应逻辑层：业务服务原子层（核心业务）
- 包含目录：`plugins/service_plugins`
- 特性：项目核心业务载体，依托总线、数据链路完成数据交互。

### 4.4 物理层4：应用编排插件（最后编译）
- 对应逻辑层：应用编排层
- 包含目录：`plugins/app_plugins`
- 特性：业务流程总调度，迭代频率最高。

### 4.5 物理层5：独立内核驱动层（完全隔离，单独编译）
- 对应范畴：Linux内核驱动（不属于应用四层逻辑架构），对应OSI硬件数据链路层
- 包含目录：`drv/`
- 编译规则：必须对接开发板Linux内核源码编译，**不纳入应用层Makefile编译体系**；
- 访问规则：应用层仅通过抽象基类+插件间接调用，禁止应用代码直接操作驱动。

### 4.6 附属资源目录（不参与编译、不纳入代码分层）
- `documents/`：项目设计文档、参考资料、调试记录；
- `output/`：编译产物、缓存文件（.gitignore 忽略）；
- `LICENSE`：开源协议文件。

## 五、核心模块职责红线（强制约束 · 严禁越界）
### 5.1 data_bus 数据总线 铁律
✅ 允许：帧句柄转发、多路由分发、事件通知
❌ 禁止：修改帧数据、格式转换、内存分配/释放、协议封装、耗时计算

### 5.2 frame_link 应用数据链路 铁律
✅ 允许：帧缓冲区分配/回收、原子引用计数管理、帧状态标记、超时释放、自定义协议封包（预留）
❌ 禁止：YUYV/RGB色彩转换、JPEG/H264编解码、图像缩放等CPU密集型运算

### 5.3 图像格式转换 & 编解码 归属标准（全局统一）
1. YUYV ↔ RGB 色彩空间转换：`common/img_proc` 公共组件（系统基建层）；
2. YUYV → JPEG 压缩：`common/img_proc` + libjpeg-turbo（系统基建层）；
3. YUYV → H264 编码：RTSP推流服务内部实现（业务服务原子层）；
4. 所有编解码、格式转换逻辑**不得移入中间枢纽层**。

### 5.4 插件体系规则
1. 所有插件统一使用 `INIT_*_EXPORT` 系列宏实现编译期自动注册；
2. 插件分为三类，职责隔离：
   - `app_plugins`：应用流程编排；
   - `base_plugins`：硬件/算法具体实现（基类子类）；
   - `service_plugins`：独立业务服务；
3. 服务插件对外接口极简，依托总线通信，无需暴露大量头文件。

## 六、标准全数据流链路（全局统一流程）
以「USB摄像头采集 → 多服务消费 → 帧回收」为例，全链路模块与层级溯源：
```
1. 摄像头硬件 → drv/内核驱动（独立硬件层）
2. 内核输出YUYV原始帧 → base_plugins/camera_usb（业务服务原子层）
3. 采集服务启动 → 向 frame_link 申请帧缓冲区、初始化引用计数（中间枢纽层）
4. 采集服务填充帧数据 → 投递帧句柄至 data_bus 转发（中间枢纽层）
5. 分支1【人脸检测】：取帧 → common/img_proc(YUYV→RGB) → MNN推理（基建层+服务层）
6. 分支2【RTSP推流】：取帧 → 服务内部H264编码 → 网络推送（服务层）
7. 分支3【SD抓拍】：取帧 → common/img_proc(JPEG压缩) → 写入存储（基建层+服务层）
8. 所有服务处理完成 → 帧引用计数归0 → frame_link 自动回收缓冲区（中间枢纽层）
```

## 七、架构演进规划（现状 + 未来扩展方向）
### 7.1 当前状态（V4.0 正式版）
1. `frame_link` 合并两大能力：帧生命周期管理（在用）+ 自定义协议封装（预留未启用）；
2. `fsm` 状态机组件代码预留，暂未落地使用；
3. 单路视频流、本地服务分发，无跨设备组网需求。

### 7.2 中期演进（遵循单一职责重构）
对 `frame_link` 做职责拆分，彻底解耦：
1. 新增 `frame_mgr` 模块：专职帧生命周期、内存池、引用计数管理；
2. 保留 `frame_link`：剥离内存管理，回归原始设计，**仅负责自定义应用协议封包/解包**；
3. 新链路：申请缓冲区(frame_mgr) → 转发(data_bus) → 协议封装(frame_link) → 消费服务。

### 7.3 远期演进（多设备/多路流场景）
1. 完善 `frame_link` 协议栈：增加分片、重传、校验、多通道标记；
2. 启用 `fsm` 状态机：管控多服务、多设备全局状态流转；
3. 扩展音频链路 `audio_link`，实现音视频一体化传输。

## 八、全局架构铁律（所有开发/评审必查）
1. **单向依赖铁律**：逻辑四层自上而下单向依赖，下层绝不反向依赖上层；
2. **通信解耦铁律**：模块间禁止直接函数调用，跨模块交互必须走事件总线/数据总线；
3. **硬件隔离铁律**：应用层不直接操作内核驱动，所有硬件能力通过「抽象基类+插件」适配；
4. **职责拆分铁律**：传输、资源管理、格式转换、编解码四类职责严格拆分，不混装在同一模块；
5. **编译隔离铁律**：`drv/` 驱动独立编译，不并入应用层编译脚本；
6. **初始化铁律**：新增模块优先使用 `initcall` 自动注册，禁止修改主函数初始化流程。
