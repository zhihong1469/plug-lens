# 结构介绍
## 架构层级:六层双总线 + 分层状态机
第一层：硬件适配层（HAL 层）
第二层：数据链路层（Link 层）
第三层：核心服务层（Service 层）
第四层：双总线中枢（Event Bus + Data Bus）
第五层：全局状态机管理层（FSM 层）
第六层：应用业务层（App 层）

> [!NOTE]
>
> 所有分层严格遵守 5 条铁律，杜绝任何藕断丝连的情况：
> 单向依赖：只能上层调用下层，下层绝对不反向依赖上层，杜绝循环依赖；
> 总线唯一通信：所有平级模块间，只能通过「双总线」通信，绝对不直接调用对方接口，彻底解耦；
> 分层状态管控：全局主状态机管控所有子模块的状态流转，子模块只能在全局状态允许的范围内运行，杜绝非法操作；
> 插件化设计：所有业务、硬件、算法能力，都以独立插件的形式存在，插拔自由，新增功能不影响原有稳定代码；
> 错误分层闭环：每一层 / 模块只处理自己职责内的错误，处理不了的仅通过总线上报，不跨层干预，不扩散故障。

## 各层**专属铁律**

### 1. HAL 层铁律（硬件抽象的底线）

1. **绝对不创建业务线程**：只做硬件操作，不参与调度
2. **绝对不碰数据流 / 业务逻辑**：只提供读 / 写 / 配置接口
3. **锁完全私有，不向上暴露**：HAL 层内部锁，上层无感知
4. **不依赖任何上层代码**：头文件不包含上层定义
5. **无状态、无全局变量、纯函数操作**：通过句柄管理状态
6. **统一错误码 + 错误描述**：便于上层统一处理异常

### 2. Link 层铁律（数据链路的底线）

1. **绝对不修改帧数据**：只做缓冲 / 路由，不做 AI 推理 / 编码
2. **绝对不直接向上层暴露帧池 / 队列 / 锁**：通过接口封装内部实现
3. **只调用 HAL 层接口**：不依赖更上层代码
4. **帧生命周期闭环**：get_frame/put_frame 严格配对，防止内存泄漏
5. **丢帧流控**：保证系统稳定性，防止缓冲区溢出
6. **按数据类型拆分链路**：视频 / 音频 / 信令链路独立，便于维护

### 3. Service 层铁律（原子能力的底线）

1. **绝对不碰硬件驱动**：只通过 Link 层获取数据
2. **服务间绝对不直接调用**：只通过总线通信
3. **不实现具体业务逻辑**：只提供原子能力（如启动 / 停止 / 配置）
4. **通用回调接口**：总线层通过回调适配，不依赖总线头文件
5. **可插拔设计**：新增服务不影响现有系统，通过总线接入
6. **状态封装**：通过不透明句柄管理内部状态，不暴露实现细节

### 4. FSM 层铁律（状态管控的底线）

1. **只做状态决策**：不做业务处理，不碰数据流
2. **分层状态管理**：通用状态抽象到父状态，具体状态在子状态实现
3. **事件驱动**：只响应总线事件，不主动发起状态变更
4. **状态机可视化**：便于调试和维护，清晰展示状态流转
5. **状态迁移原子性**：防止状态不一致，保证系统稳定性

### 5. App 层铁律（业务调度的底线）

1. **只做业务组合**：通过调用 FSM 层和 Service 层接口实现业务逻辑
2. **用户交互隔离**：UI 操作只通过 App 层处理，不渗透到下层
3. **配置集中管理**：系统参数统一配置，不分散到各层
4. **错误统一处理**：捕获下层错误，统一日志 / 告警 / 恢复机制

### 6. 双总线铁律（横向通信的底线）

1. **总线只做路由转发**：不做任何业务处理
2. **绝对不允许模块间绕过总线直接通信**：保证通信可追溯、可管控
3. **Event Bus 轻量**：只传递控制指令、状态变化、异常告警等小数据
4. **Data Bus 零拷贝**：传递视频帧、音频帧等大数据时，只传指针和元信息
5. **总线接口标准化**：所有模块通过统一接口接入，便于扩展
6. **总线可监控**：支持事件 / 数据统计、调试、性能分析

## 总结：架构的 “灵魂” 与 “骨架”

1. **灵魂**：**单向依赖 + 职责单一**，确保系统可维护、可扩展
2. **骨架**：**六层分层 + 双总线**，提供清晰的边界和通信机制
3. **肌肉**：各层的严格铁律，保证系统稳定性和可靠性

> **松耦合**：模块独立升级 / 替换，不影响其他模块
>
> **高内聚**：每层只做自己该做的事，职责清晰
>
> **可测试**：各层可独立单元测试，提高代码质量
>
> **可扩展**：新增功能只需添加新模块，不修改现有代码
