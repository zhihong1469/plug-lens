# 安全红线规则

## 1. 备份规则

### 1.1 备份存储位置
- **禁止**将备份文件仅保存在远程设备（开发板）的临时目录（如 `/tmp`）
- **必须**将备份文件同步到本地主机持久化存储
- 备份文件路径格式：`{project}/board/{platform}/backup/{date}/{type}.img`
- 每次操作前必须创建新的日期目录，避免覆盖历史备份

### 1.2 备份验证
- 备份完成后必须验证文件完整性（MD5/SHA256 校验）
- 必须确认本地备份文件存在后，才能执行破坏性操作
- 校验和文件必须与备份文件保存在同一目录下

### 1.3 备份保留
- 备份文件至少保留 30 天
- 在确认新固件稳定运行前，不得删除备份
- 保留至少3个历史版本的备份

## 2. 固件烧录规则

### 2.1 烧录前检查清单
- [ ] 本地备份文件已存在并验证完整
- [ ] 备份文件大小正确
- [ ] 恢复方案已准备就绪
- [ ] 交叉编译工具链已正确配置
- [ ] 目标设备型号确认无误

### 2.2 烧录操作规范
- 烧录前必须向用户确认操作内容和风险
- 烧录后必须验证设备能否正常启动
- 如有异常，立即执行恢复流程
- 新固件验证稳定运行72小时后，才能清理旧备份

### 2.3 恢复方案
- 必须准备至少一种恢复方法（SD卡、串口、MaskROM等）
- 恢复工具和文档必须提前准备并测试可用
- 必须在操作前验证恢复方法有效

## 3. 远程设备操作规则

### 3.1 破坏性操作定义
- dd 写入分区
- 格式化存储设备
- 修改引导配置（bootloader、bootargs等）
- 更新内核/设备树
- 修改U-Boot环境变量

### 3.2 操作流程
1. 先备份到远程设备
2. **立即**同步备份到本地
3. 验证本地备份完整性（MD5校验）
4. 确认用户已知晓操作风险
5. 执行破坏性操作
6. 验证操作结果
7. 记录操作日志

### 3.3 禁止操作
- 禁止在没有本地备份的情况下执行破坏性操作
- 禁止在用户未确认的情况下执行破坏性操作
- 禁止跳过验证步骤直接执行破坏性操作
- **禁止直接修改内核源码中的 .dtsi 文件**（如 rk3562.dtsi），必须通过板级dts或overlay修改

## 4. 设备树修改规则

### 4.1 修改原则
- **最小修改原则**：只修改必要的属性（如 `status`），不碰 `reg`、`interrupts`、`clocks` 等关键配置
- **标签引用原则**：使用 `&label` 语法引用现有节点，不重新定义节点
- **成品镜像优先**：能用成品镜像改就绝不重新编译内核

### 4.2 禁止操作
- 禁止直接修改芯片级 `.dtsi` 文件（如 `rk3562.dtsi`）
- 禁止重写整个设备树节点（会覆盖原厂配置）
- 禁止修改 `reg`、`interrupts`、`clocks`、`power-domains`、`pinctrl` 等硬件相关属性

### 4.3 正确修改方法
- 修改板级 `.dts` 文件（如 `rk3562-xxx.dts`）
- 使用标签引用语法：`&npu { status = "okay"; };`
- 通过设备树Overlay（DTBO）动态修改

## 5. 示例：正确的备份流程

```bash
# 错误示例 ❌
ssh root@device "dd if=/dev/mmcblk0p3 of=/tmp/backup.img"
# 备份仅存在于远程设备，断电即丢失

# 正确示例 ✅
# 1. 创建日期目录
mkdir -p ./board/rk3562/backup/$(date +%Y%m%d)

# 2. 备份到远程设备
ssh root@device "dd if=/dev/mmcblk0p3 of=/tmp/boot_backup.img"

# 3. 立即同步到本地
scp root@device:/tmp/boot_backup.img ./board/rk3562/backup/$(date +%Y%m%d)/boot.img

# 4. 生成校验和
md5sum ./board/rk3562/backup/$(date +%Y%m%d)/boot.img > ./board/rk3562/backup/$(date +%Y%m%d)/checksum.md5

# 5. 验证本地备份
ls -lh ./board/rk3562/backup/$(date +%Y%m%d)/
cat ./board/rk3562/backup/$(date +%Y%m%d)/checksum.md5

# 6. 确认后再执行破坏性操作
```

## 6. 示例：正确的设备树修改流程

```bash
# ❌ 错误：直接修改.dtsi文件
vim kernel-6.1/arch/arm64/boot/dts/rockchip/rk3562.dtsi  # 禁止！

# ✅ 正确：修改板级.dts文件
vim kernel-6.1/arch/arm64/boot/dts/rockchip/rk3562-xxx.dts

# 在文件末尾添加：
&npu {
    status = "okay";
};

&rknn_mmu_0 {
    status = "okay";
};
```
