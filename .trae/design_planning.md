# `.trae` 技能体系设计规划文档
> **文档定位**：本文件是本项目 Trae 技能体系的总设计纲领与最高规范，所有 Agent 对 `.trae` 目录的构建、修改、扩展、重构操作，必须严格遵循本文档约定。任何与本文档冲突的结构均视为不规范，需按本文档对齐。

---

## 一、设计目标
基于线束工程（Harness Engineering）方法论，构建一套**分层复用、渐进迭代、稳定可控**的嵌入式 Linux 开发 AI 协作体系，实现以下目标：
1. **约束标准化**：所有代码输出统一遵循嵌入式编码规范与平台约束，输出稳定可预期
2. **能力模块化**：专项开发能力封装为独立可插拔技能包，按需加载、跨项目复用
3. **演进自动化**：Agent 可基于本规范自行维护、扩展、优化技能体系，无需人工干预结构设计
4. **安全边界化**：内置多层安全护栏，杜绝危险操作与不符合工程规范的输出

---

## 二、整体架构设计
采用三层平级架构，自下而上分别为：**基础约束层 → 专项能力层 → 输出标准层**，三层职责独立、单向依赖、不可反向嵌套。

| 层级 | 对应目录 | 核心定位 | 生效方式 | 优先级 |
|------|---------|---------|---------|--------|
| L1 基础约束层 | `rules/` | 规定「必须遵守什么、绝对不能做什么」，是所有输出的底线规范 | 项目全局永久生效 | 基础层，所有技能必须遵守 |
| L2 专项能力层 | `skills/` | 封装「某一类任务的完整解决方案」，提供工具、流程、参考资料 | 按需触发加载，对应场景自动激活 | 扩展层，在规则约束内叠加能力 |
| L3 输出标准层 | `templates/` | 规定「输出长什么样」，统一代码骨架与交付格式 | 生成对应类型代码时自动参考 | 标准层，提升输出一致性 |

### 架构原则
- **平级关系**：`rules`、`skills`、`templates` 为同级目录，互不嵌套，均直属 `.trae/` 根目录
- **单向依赖**：技能可以引用规则与模板，规则与模板不得依赖具体技能
- **分层复用**：通用规则下沉，专项能力上浮，避免重复定义
- **渐进扩展**：优先复用现有能力，确有不足再新增文件，不追求大而全

### 目录职责说明
| 路径 | 职责 | 维护规则 |
|------|------|---------|
| `rules/` | 存放所有全局生效的规则文件 | 按数字前缀分层，数字越小越通用，数字越大优先级越高；支持三级子目录嵌套 |
| `skills/` | 存放各独立技能包，每个子目录为一个完整技能 | 每个技能必须包含 `SKILL.md` 作为官方识别入口；共享工具统一放入 `shared/` 目录 |

| `design_planning.md` | 本设计文档 | 结构变更必须先更新本文档，再执行修改 |

## 四、核心设计原则
### 1. 分层复用原则
- 通用规范下沉至 `00-` 开头的基础规则，跨项目可直接复用
- 平台专属内容放入 `01-` 层，同芯片平台可复用
- 项目特有内容放入 `02-` 层，仅当前项目生效
- 安全红线独立为最高层级，所有场景强制生效

### 2. 最小化原则
- 一个文件只负责一个主题，禁止编写大而全的单体规则文件
- 一个技能只解决一类问题，职责边界清晰，不做能力溢出
- 能复用现有能力就不新增，避免冗余与不一致

### 3. 渐进式迭代原则
- 技能集不是一次性写完的，随开发过程逐步完善
- 每遇到一次 AI 犯错，就补充一条对应规则
- 每完成一次重复流程，就沉淀为一个技能模板
- 不追求一次性完美，小步快跑、持续优化

### 4. 可执行性原则
- 所有规则必须明确、可落地，禁止「写好代码」「注意质量」等模糊表述
- 所有技能必须包含可执行的步骤或脚本，不能只有概念描述
- 所有安全红线必须有明确的判定标准，可自动检查

## 五、规则文件（rules）编写规范
### 1. 命名规范
- 强制使用 `数字前缀-英文标识.md` 格式
- 数字前缀范围 `00-99`，数字越大优先级越高
- 英文标识使用小写+连字符（kebab-case）
- 示例：`00-embedded-base.md`、`01-rk3562-hardware.md`、`99-security-redline.md`

### 2. 优先级规则
1. 高优先级文件覆盖低优先级文件的同名配置
2. `99-` 开头的安全红线拥有最高优先级，不可被任何其他规则覆盖
3. 子目录内的规则继承父目录优先级，同级文件按文件名排序

### 3. 标准文件结构
每个规则文件必须遵循以下结构：
```markdown
# 规则名称
## 一、适用场景
明确本规则在什么情况下生效

## 二、核心规范
分点列出明确的、可执行的约束，每条只讲一件事
✅ 正面示例：寄存器操作后必须添加内存屏障 dmb()
❌ 反面示例：注意寄存器操作的正确性

## 三、禁止事项
明确列出绝对不允许的行为

## 四、参考示例（可选）
正确与错误的代码对比
```

### 4. 编写要求
- 使用第二人称指令语气，直接告诉 AI 要做什么
- 多用肯定句，少用疑问句和模糊表述
- 关键术语加粗，重要编号有序排列
- 嵌入式相关规则必须标注硬件约束来源

---

## 六、技能包（skills）编写规范
### 1. 技能标准结构
每个技能为一个独立文件夹，必须包含以下内容：
```
skill-name/
├── SKILL.md        # 必需：技能入口与核心指令
├── scripts/        # 可选：可执行脚本（Python/Shell）
├── references/     # 可选：参考文档、手册、数据表
├── agents/         # 可选：Agent元数据配置
└── examples/       # 可选：输入输出示例
```

### 2. SKILL.md 强制格式
必须包含 YAML 头信息 + Markdown 正文，格式如下：
```markdown
---
name: build-linux-app
description: "自动识别Linux应用项目，调用交叉编译器完成编译构建、产物定位与错误分析。触发关键词：编译、构建、build、make"
---

# Linux 应用构建技能
## 适用场景
- Linux 嵌入式应用程序交叉编译
- Makefile / CMake 项目构建
- 编译错误排查与修复建议

## 核心能力
1. 自动识别项目构建系统
2. 自动匹配交叉编译工具链
3. 执行构建命令并捕获输出
4. 分析编译错误并给出修复方案

## 执行流程
1. 探测项目根目录构建文件
2. 匹配对应构建工具与参数
3. 执行编译命令
4. 解析输出，成功则返回产物路径，失败则输出错误分析

## 依赖工具
- aarch64-none-linux-gnu-gcc
- make / cmake
- 共享库：skills/shared/tool_config.py

## 输出规范
- 结构化输出：状态 + 产物路径 + 耗时 + 错误详情
- 编译失败必须定位到具体文件与行号
```

### 3. 命名规范
- 技能文件夹名：小写+连字符，动宾结构，如 `build-linux-app`、`debug-gdb`
- `name` 字段与文件夹名保持一致
- `description` 字段必须包含触发关键词，提升技能召回准确率

### 4. 共享库规范
- 多个技能共用的工具函数放入 `skills/shared/` 目录
- 共享代码必须跨平台兼容，遵循统一的工具探测优先级
- 修改共享代码必须同步验证所有依赖技能的可用性

---

## 七、模板与参考资料规范
### 1. 代码模板（templates）
- 仅存放标准骨架代码，不包含业务逻辑
- 必须包含完整的头文件引用、错误处理、资源释放结构
- 关键位置添加注释占位符，标注需要填充的内容
- 命名格式：`场景-功能.后缀`，如 `rknn-inference.c`

### 2. 参考资料（references）
- 放在对应技能内部，不放到顶层
- 优先使用 Markdown 格式，便于 AI 读取解析
- 数据表、手册等大文件添加摘要说明，避免占用过多上下文

## 八、Agent 自我迭代工作流
> Agent 在执行技能集自我完善任务时，必须严格遵循以下流程，禁止随意改动目录结构。

### 流程1：新增一条规则
1. 判断规则所属层级：通用基础 / 平台专属 / 项目特有 / 安全红线
2. 对应到 `rules/` 下对应前缀的文件
3. 若现有文件主题匹配，则追加到对应章节
4. 若主题不匹配，新建对应前缀的规则文件
5. 保持命名规范与文件结构一致

### 流程2：新增一个技能
1. 确认现有技能无法覆盖需求，确有新增必要
2. 在 `skills/` 下新建技能文件夹，命名遵循规范
3. 创建标准 `SKILL.md` 入口文件，填写完整元数据
4. 按需添加 scripts / references / examples
5. 若有可复用逻辑，优先引用 `skills/shared/` 共享库

### 流程3：优化现有内容
1. 只修改内容，不轻易改动文件结构与命名
2. 修改规则必须保持格式规范统一
3. 修改技能必须更新 `SKILL.md` 描述与能力说明
4. 涉及共享库修改，必须同步检查所有依赖技能

### 流程4：结构重构
1. **结构重构必须先更新本文档 `design_planning.md`**
2. 说明重构原因、新结构设计、迁移方案
3. 经确认后再执行目录调整
4. 重构后验证所有技能与规则正常生效

---

## 九、全局安全红线
所有规则、技能、Agent 操作必须遵守以下底线，任何情况下不得突破：
1. **禁止危险命令**：不得自动执行 `rm -rf`、`dd`、`flash`、烧录固件等破坏性操作，必须人工确认
2. **禁止越权操作**：不得自动执行 `sudo`、修改系统配置、修改内核参数等特权操作
3. **禁止无依据硬件修改**：寄存器配置、设备树修改必须标注依据，提醒用户对照手册审核
4. **禁止静默失败**：所有错误必须明确抛出，不得掩盖报错、忽略异常
5. **禁止脱离规范**：所有输出必须遵循既定编码规范与安全约束，不得自行其是

## 十一、跨平台工作流设计

### 场景定义

| 环境 | 角色 | 主要职责 |
|------|------|---------|
| **Windows** | USB 转发层 | 运行 `usbipd` 命令，将 USB 设备转发到 WSL2 |
| **WSL2 (Linux)** | 主开发环境 | 编译、调试、串口通信、项目开发 |

### 推荐工作流程

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1 [Windows 管理员终端]                                │
│    python scripts/windows/wsl2_usb_forward.py auto         │
│    → 自动检测 CH340 (VID/PID=1a86:7523) 并转发到 WSL2     │
├─────────────────────────────────────────────────────────────┤
│  Step 2 [WSL2 终端]                                        │
│    python scripts/linux/serial_tool.py list-ports          │
│    → 验证串口 /dev/ttyUSB0 是否已转发成功                  │
├─────────────────────────────────────────────────────────────┤
│  Step 3 [WSL2 终端]                                        │
│    python scripts/linux/serial_tool.py login-check         │
│    → 连接开发板并执行验证命令                              │
├─────────────────────────────────────────────────────────────┤
│  Step 4 [WSL2 终端]                                        │
│    python build-linux-app/scripts/linux_builder.py         │
│    → 交叉编译应用程序                                      │
├─────────────────────────────────────────────────────────────┤
│  Step 5 [WSL2 终端]                                        │
│    python scripts/linux/serial_tool.py login-check         │
│      --cmd "cd /mnt/nfs && ./myapp"                       │
│    → 在开发板上运行程序并验证                              │
└─────────────────────────────────────────────────────────────┘
```

### 串口设备识别策略

| 场景 | 识别方式 | 稳定性 |
|------|---------|--------|
| Windows 直连 | COM 端口号（可能变化） | 低 |
| WSL2 转发 | VID/PID 匹配（固定） | 高 |
| WSL2 访问 | `/dev/ttyUSB0`（转发后固定） | 高 |

> **关键设计**：使用 VID/PID（如 CH340 的 `1a86:7523`）作为稳定标识，避免 COM 端口号动态变化的问题。
