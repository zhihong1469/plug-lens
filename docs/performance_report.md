## 🚀 v0.9.0 功能调试版（2026-05-20）
### 版本状态
**单功能可用，多功能冲突严重**：人脸抓拍功能完全正常，但帧率过高导致CPU占用过大系统卡顿；RTSP推流存在严重花屏、断流、数据丢失问题；多功能同时运行时推流完全卡死，仅抓拍可运行。

### 性能基准测试（原始调试数据）

| 运行场景 | 进程CPU占用 | 系统总CPU占用 | 内存占用(VSZ) | 状态说明 |
|---------|------------|--------------|--------------|----------|
| 仅视频采集 | 30% | 45% | 85MB | 运行正常 |
| 采集+RTSP推流 | 85% | 92% | 85MB | 推流严重花屏、断流，且帧率<3fps |
| 采集+人脸检测 | 95% | 100% | 85MB | 抓拍正常，系统卡顿 |
| 采集+人脸检测+RTSP推流 | 97% | 100% | 85MB | 推流显示一帧后卡死，抓拍正常 |
> 测试方式：无智能调度，所有线程开启拿帧就跑
### 存在的核心问题（实事求是记录）
1. **无资源调度机制**：所有线程无限制全速运行，CPU持续满载
2. **帧率严重不匹配**：采集30fps，AI推理仅1fps，产生29帧无效数据浪费CPU
3. **推流链路不稳定**：H264编码参数不匹配，跳帧过多;帧缓存溢出导致数据丢失、花屏断流
4. **无模式隔离**：多功能同时运行时推流常被满载卡顿打断黑屏
5. **无休眠机制**：无客户端连接时推流服务仍全速编码，浪费CPU

### 未观测到严重内存泄漏
调试阶段未进行超长时间内存监控，仅观测到单功能运行时内存波动在正常范围内，无明显OOM崩溃现象。

## ✅ v1.0.0 正式稳定版（2026-05-29）
### 版本状态
**工业级场景基本可用，达到i.MX6ULL硬件性能极限**：人脸抓拍开销可控，符合工业场景无人值守需求；RTSP推流流畅度大幅提升，7fps稳定输出，极少出现花屏断流；全流程稳定运行，可长期部署。

### 测试环境说明
- 系统：Linux 4.9.88 + 定制裁剪Buildroot
- 测试工具：`top -b -n 1` 实时采样 + 系统内置分级日志
- 测试时长：连续运行18小时（17:52-次日11:52）
- 测试场景：覆盖系统启动、人脸抓拍、RTSP推流、智能模式切换全流程
- 基准线：系统空载CPU占用 < 5%，内存占用 < 50MB

### 系统启动流程分析
```
[0ms] 初始化TLSF静态内存池 (40MB)
[12ms] 初始化事件总线 + 数据总线
[25ms] 发布SYS_CORE_READY事件
[30ms] 启动优先级自动初始化(InitCall机制)
  - Priority 2: 视频采集服务 (640x360@30FPS, 软件降频至14FPS)
  - Priority 3: RTSP推流服务 + 人脸检测服务(14/5fps) + 图片存储服务
  - Priority 4: 应用编排服务
[120ms] 所有模块初始化完成
[150ms] 发送系统启动指令，各服务自动启动
[200ms] RTSP服务启动成功: rtsp://192.168.5.9:8554/stream
[250ms] 系统进入无人值守运行模式
```

### 核心功能运行日志
#### 1. 智能模式切换（核心优化点）
```
[RTSP] 客户端连接 | 在线数: 1
[NET_PUSH] RTSP客户端已连接，暂停人脸抓拍
[FACE_DETECT] RTSP推流中，暂停人脸检测/抓拍

[RTSP] 客户端断开 | 在线数: 0
[NET_PUSH] RTSP客户端已断开，恢复人脸抓拍
[FACE_DETECT] RTSP已断开，恢复人脸检测/抓拍
```
**设计思路**：基于工业场景需求设计，优先保证单一核心功能稳定：
- 无人查看时：后台静默人脸抓拍，低功耗运行
- 有人查看时：暂停抓拍，全力保证视频流流畅度

#### 2. 人脸抓拍功能
```
[IMG_STORAGE] JPG saved: face_20260529_172643.jpg | Size:27819 Bytes
[IMG_STORAGE] JPG saved: face_20260529_172645.jpg | Size:28705 Bytes
...
```
- 平均每张人脸图片大小：~28KB
- 抓拍频率：14/5fps（可结合场景配置）
- 存储系统：全资源开销可控,设置滚动覆盖

#### 3. RTSP流媒体服务
```
[NET_PUSH] 创建H264编码器 | 640x360 | 14FPS | GOP=14
[NET_PUSH] 获取SPS+PPS成功 | 大小: 27 bytes
[NET_PUSH] RTSP服务启动成功
```
- 支持单客户端同时拉流
- 自动检测客户端状态，异常断开自动清理资源
- 推流帧率稳定在7fps（i.MX6ULL软编码物理极限）
- 花屏断流概率<1%，工业场景可接受

### 资源消耗基准测试（优化后真实数据）
| 运行场景 | 进程CPU占用 | 系统总CPU占用 | 内存占用(VSZ) | 系统总内存占用 | 状态说明 |
|---------|------------|--------------|--------------|----------------|----------|
| 系统空载（无应用） | - | < 5% | - | < 50MB | 系统基准 |
| 应用启动完成（无服务） | 0% | < 10% | 93MB | ~197MB | 所有服务休眠 |
| **人脸抓拍模式（无推流）** | **平均8%** | **平均25%** | 93MB | ~200MB | ✅ 推荐日常无人值守模式 |
| **RTSP推流模式（无抓拍）** | **平均80%** | **平均90%** | 93MB | ~200MB | ✅ 推荐临时查看模式 |
| 全功能同时运行 | 95% | 100% | 93MB | ~200MB | ⚠️ 不推荐，推流极易卡顿 |
| **连续运行18小时** | 平均8% | 平均25% | 93MB | ~245MB | ✅ 无内存泄漏，运行稳定 |

### 关键发现
1. **内存极度稳定**：应用进程内存始终稳定在93MB，18小时无内存泄漏
2. **长时间运行可靠**：实测18小时系统总内存增长48MB（全部为Linux系统文件缓存）
3. **智能调度效果显著**：人脸抓拍模式CPU占用从90%降至25%，优化幅度72%
4. **推流稳定性大幅提升**：解决了花屏、断流、数据丢失问题，达到工业可用标准
5. **已达硬件极限**：RTSP推流模式CPU占用90%，无进一步大幅优化空间
>[!NOTE]
> 注意:这里人脸抓拍模式的CPU占用并非一直在跑,25%是指有人脸活动状态下的14/5fps的平均开销,高峰值一般为85%+的CPU开销,不可避免!
### 已实现的核心优化成果
#### 1. 智能动态调度（人脸抓拍CPU下降72%）
- **无人流自动休眠**：RTSP无客户端时，停止H264编码，CPU从85%降至8%
- **AI推理帧率匹配**：采集降频至5fps，与AI推理速度精准对齐，消除无效帧
- **模式互斥隔离**：推流与抓拍互斥运行，避免资源抢占导致的卡顿
- **线程优先级分级**：推流线程(90) > 采集线程(80) > AI推理线程(70)，保证核心功能优先

#### 2. 推流稳定性修复
- 优化H264编码参数，调整GOP大小与帧缓存深度
- 实现帧丢失自动重传机制，解决花屏问题
- 增加RTSP客户端心跳检测，异常断开自动清理资源

#### 3. 内存管理优化
- **TLSF静态内存池**：40MB预分配，运行期零malloc，杜绝内存碎片
- **帧引用计数机制**：多模块共享同一帧数据，无内存拷贝
- **自动帧回收**：超时未处理的帧自动释放

### 稳定性说明
- 实测连续运行18小时无崩溃、无内存泄漏
- 结合系统看门狗机制，推测可支持7*24小时无人值守运行（未进行极限测试）
